# Cross-Question LongMemEval-S First Baseline (M2)

## Run

Command:

```bash
rtk node apps/bench-runner/bin/alaya-bench-runner.mjs longmemeval-crossquestion \
  --variant s --limit 50 --embedding disabled \
  --history-root docs/bench-history
```

- Archive: `docs/bench-history/public-crossquestion/2026-05-15T135017Z-af4a721/`
- Sample: 50 questions in **one shared workspace, one shared daemon, one
  run**. Each question's haystack is seeded incrementally; previous
  questions' memories and RECALLS edges remain in the pool.
- Final shared pool size: **25,024 memories** (vs ~500 per question in
  single-turn isolated harness — a ~50× expansion).
- Embedding: disabled (heuristic-removed `af4a721` build).

| Surface | n | Pool size | R@5 | p95 |
|---|---:|---:|---:|---:|

The multi-turn 84% looked dramatic, but on the **same first-50 sample
in a fresh-per-question workspace** it is now revealed as
sample-skewed: the cross-question harness on the same first-50
questions hits 62.0% — multi-turn's headline number was not
generated by structural recall improvement, it was the easier
first-50 subset of the dataset.

## Per-plane evidence under a real shared workspace

`plane_winning_admission` distribution across the 51 gold pointers
(50 questions, 1 multi-gold question contributes 2):

| Plane | Single-turn 100 | Multi-turn 50×3 (round 3) | **Cross-question 50** |
|---|---:|---:|---:|
| `session_surface_cohort` | 94% | 96% | **86% (43/50)** |
| `evidence_anchor` | 2% | 2% | **12% (6/50)** |
| `lexical` | 3% | 0% | 2% (1/50) |
| `graph_expansion` | 0% | 4% | **0%** |
| `path_expansion` | 0% | 0% | 0% |
| `activation` | 1% | 0% | 0% |

**Three observations:**

1. **`evidence_anchor` finally shows up** — from ~2% in single-turn
   to **12% in cross-question**. In a shared workspace where each
   question's seeds carry question-prefixed `evidence_refs`, the
   plane gets a real differentiation signal and rescues 6 gold
   pointers that would otherwise be lost. This is the first
   architectural plane other than lexical/cohort that empirically
   moves the hit rate on this surface.

2. **`graph_expansion` still does nothing** — 0 winning admissions
   even with 50 sequential `report_context_usage` calls writing
   RECALLS edges between them. The cross-link `< 2 used objects`
   guard combined with mostly-single-gold LongMemEval questions
   means RECALLS edges that do get written rarely connect to the
   *next* question's gold (they connect within a question's used
   set).

3. **`session_surface_cohort` is still the dominant plane (86%)**.
   In a shared workspace with 25,024 memories, the cohort fan-out
   cap (`DYNAMIC_RECALL_PLANE_CAP = 240`) does kick in, so it is no
   longer literal-everything; but for a given question's recall it
   still over-admits aggressively because nothing else in the
   feature set distinguishes "memories from this question" from
   "memories from other questions" — they all carry
   `domain_tags: ["bench-seed"]` and live in the same workspace.

## Why didn't accumulation help last-half?

`last_half (60%) < first_half (64%)`. Middle bucket is the real
dip (57%). Breakdown of misses:

| Miss class | First 10 | Middle 30 | Last 10 |
|---|---:|---:|---:|
| `budget_dropped` | 3 | 8 | 2 |
| `under_ranked` | 0 | 5 | 0 |
| `candidate_absent` | 0 | 0 | 1 |

Root causes for the no-accumulation result:

- **Each question's query text shares no terms with previous
  questions' used gold** (different topics, different users in the
  dataset). Even if RECALLS edges existed between previous useds,
  the current question's recall does not seed from those nodes —
  seeds come from the activation + lexical match of the current
  query, which is unrelated.
- **`crossLinkRecalledMemories` requires ≥ 2 used objects** in a
  *single* `report_context_usage` call. 49/50 questions still have
  exactly 1 used gold, so RECALLS edges are still essentially
  never written across the run.
- **Score-flattening makes large pool worse, not better.** With
  25,024 memories competing, lexical hits in unrelated questions'
  haystacks pull score down on the actual gold's neighborhood. The
  baseWeight=0.52 + relevance weight 0.34 weighting means
  `activation * 0.52` dominates a flat-scored pool — and other
  questions' memories have comparable activation (same
  `bench-seed` provenance), so they noise into the top10.
- **Latency cost is real**: p95 jumped from ~150ms to 4065ms
  because the multi-plane scanner reads 25K rows for every query.

## Verdict — Multi-axis architecture is real, but currently nets *negative* on shared workspaces

This is the strongest evidence we have so far that the multi-plane
architecture **does have a real differentiation signal** —
`evidence_anchor` goes from invisible (2%) to load-bearing (12%) the
moment a shared workspace gives it real cross-question
differentiation to do. That refutes the earlier hypothesis "multi-
plane is decorative on this dataset."

But it also reveals that today the architecture **net loses** in
shared-workspace conditions:

- Pool grows 50× → R@5 drops 7 pp (69% → 62%)
- `session_surface_cohort` dominates winning admissions, behaving
  like a "same-question fallback" rather than a structural plane
- `graph_expansion` and `path_expansion` contribute zero on this
  bench because of the cross-link guard and the question-
  independence of LongMemEval's user queries
- Latency p95 grows 25×

## Implications for Phase C — high-leverage, all general

The cross-question evidence sharpens Phase C priorities:

### C-1: Widen the delivery budget (highest leverage, do first)

`packages/core/src/task-surface-builder.ts:74` — `chat.fine.budgets.max_entries: 10 → 15` (analyze/build/govern in
proportion). 13 of 50 cross-question questions are `budget_dropped`;
of those, most have gold in pre_budget_rank 11-20. Widening alone
could reclaim 5-10 gold and push R@5 toward 70-80% without changing
any plane logic. This is the cleanest non-overfit improvement
available, and it does NOT depend on Phase B / M2 verdict.

### C-2: Demote `session_surface_cohort` to advisory when its admission ratio is high

`packages/core/src/recall-service.ts:664-697` — when
`exactCohortMatches.length / tierMemories.length > 0.5`, set
`isAdvisory=true`. On small per-question workspaces (single-turn,
multi-turn isolated) this stops cohort from monopolizing winning
admission. On large shared workspaces it cleans up the noise floor
so `evidence_anchor` and `domain_tag_cluster` can compete for
winning admission slots they currently lose.

### C-3: Fill in seed metadata so evidence_anchor / domain_tag_cluster scale further

`apps/bench-runner/src/harness/daemon.ts:430-438` — replace fixed
`domain_tags: ["bench-seed"]` with `["bench-seed", "session:<sessionId>",
"qid:<question_id>"]`. The cross-question harness already provides
the surface where `evidence_anchor` works (12% winning admission);
adding multi-element tags should let `domain_tag_cluster` join it.
This is bench-harness only — does not pollute core with
question-shape rules.

### C-4: Investigate `crossLinkRecalledMemories` ≥ 2 guard

`apps/core-daemon/src/mcp-memory-tool-handler.ts:1144` — the guard
is structurally correct (a single used object cannot form a graph
edge), but it means most LongMemEval questions never write any
RECALLS edge. Options worth evaluating in v0.3.8:
- write edges between `used` and other co-delivered (but not used)
  candidates with a weaker edge type (e.g. `CO_DELIVERED`), so a
  graph signal accumulates even from single-used recalls
- or accept that RECALLS-edge accumulation needs multi-gold or
  multi-step agent flows to develop, and document that LongMemEval
  is not the surface where graph_expansion proves value.

### C-5: Latency regression budget on shared pools

p95 4065ms at 25K memories is a real production concern. If
shared workspaces grow this large, the multi-plane scanner needs
either tier filtering applied earlier or a per-plane budget cap
that scales with pool size rather than a fixed `PLANE_CAP=240`.
Tracked in Phase D candidate.

## What to do next

Phase C-1 first (independent and high-leverage). Then C-2 + C-3
together (they share the same goal: let evidence-based planes
compete with cohort). Re-run all three bench surfaces (single-turn
100, multi-turn 50×3, cross-question 50) after each Phase C change
and update this report.

Phase C-4 (cross-link guard) is research, not bench-driven; defer
to v0.3.8 unless we see something surprising.
