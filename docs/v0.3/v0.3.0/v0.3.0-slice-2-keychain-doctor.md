## 1. Background & Goal

**Background.** `apps/core-daemon/src/cli/doctor.ts` builds a
fixed-shape `DoctorReport` whose `checks` record has a fixed key set
(`"runtime" | "storage" | "provider" | "mcp" | "garden"`), with all
external probes injected as optional dependencies — when a probe is
omitted the report falls back to a conservative snapshot (the
`getGardenCompute` / `credential_source` precedent). After
v0.3.0-slice-1, a runtime config can carry a
`keychain:<service>:<account>` `secret_ref`, but `doctor` does not
yet tell the operator whether that ref actually resolves on this
host.

**Goal.** When a `keychain:` `secret_ref` is configured, `alaya
doctor` verifies the host-platform adapter is callable and the entry
resolves, surfacing a clear remediation message otherwise; when no
`keychain:` ref is configured, the keychain check is a silent no-op
and the report (including `checks`) is byte-identical to current
behaviour for `env:` / `file:` installs.

## 2. Allowed Scope

### apps/core-daemon/src/cli/doctor.ts

- **Change**: add a keychain-resolution probe. When the resolved
  runtime config's `secret_ref` (the same one `doctor` already reads
  for `provider` / `garden_compute`) starts with `keychain:`, call
  `resolveSecretRef` (via the same `SecretRefReader` path slice 1
  established) and:
  - resolves to a value → report `keychain_ok: true` (or an
    equivalently named field on the relevant section — `provider` /
    `garden_compute`; reuse the section that already owns
    `secret_ref` health, do not invent a new top-level `checks` key);
    the section's `checks` status is unaffected (still `pass` if
    everything else is fine).
  - tooling-not-available `ResolveSecretError` → remediation string
    naming the platform tool and the package to install (Linux:
    `libsecret-tools`; macOS: `security` is built in — phrase
    accordingly; Windows: the credential-manager prerequisite from
    slice 1's decided mechanism); the relevant `checks` entry goes
    `fail` (or `degraded`, matching how `doctor` already grades a
    broken `secret_ref`).
  - entry-not-found `ResolveSecretError` → remediation string naming
    the `service` / `account` the operator must populate (never the
    secret value); `checks` entry `fail` / `degraded` as above.
  - **no `keychain:` ref configured** → the probe does nothing; the
    report and `checks` are unchanged from pre-card behaviour.
- **Change**: include the new field in the `--json` output with a
  stable shape (optional / present-only-when-relevant; document the
  shape in a code comment anchor).

### apps/core-daemon/src/index.ts

- **Change**: only if the keychain probe needs a new injectable
  dependency on `DoctorCommandDependencies` (e.g.
  `getSecretRefKeychainStatus`), wire it the same way the existing
  optional probes are wired, with a clean "dependency omitted →
  conservative no-op" branch. If the probe can reuse the
  config/reader plumbing `doctor` already has, do not add a new
  dependency.

### apps/core-daemon/src/__tests__/<doctor test file>

- **Change**: add cases — (a) `keychain:` ref + adapter callable +
  entry present → `keychain_ok: true`, `checks` still `pass`; (b)
  adapter callable, entry missing → remediation names `service` /
  `account`, no secret value in the output, `checks` entry
  `fail`/`degraded`; (c) adapter binary missing → remediation names
  the package/prerequisite, `checks` entry `fail`/`degraded`; (d)
  **no `keychain:` ref → `DoctorReport` and `checks` byte-identical
  to the pre-card baseline** (regression guard — snapshot or
  deep-equal against the env/file fixture); (e) `--json` output
  includes the new field with the stable shape; (f) the new
  dependency (if added) omitted → conservative no-op, no new `fail`.

### docs/v0.3/v0.3.0/reports/v0.3.0-slice-2.md (new)

- **Change**: completion report.
