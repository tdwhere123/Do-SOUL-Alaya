## 1. Background & Goal

**Background.** `apps/core-daemon/src/cli/install.ts` already owns
the onboarding pipeline — it reads an existing install config,
resolves a final config, reads `ALAYA_OPENAI_SECRET_REF` from an env
file, manipulates TOML sections, and writes the runtime config. After
v0.3.0-slice-1, `secret_ref` can be a `keychain:<service>:<account>`
ref, but there is no operator-facing path to *create* one — the
operator would have to hand-edit config and hand-store the secret.

**Goal.** Add an `alaya install --keychain` flag that prompts the
operator for the secret value, writes it into the host platform
keychain via the adapter, and rewrites the runtime config `secret_ref`
to `keychain:<service>:<account>` — leaving the prior `env:` / `file:`
ref on disk so the operator reverts by reverting the config patch.

## 2. Allowed Scope

### apps/core-daemon/src/cli/install.ts

- **Change**: add a `--keychain` flag. Interactive flow:
  1. Determine `service` / `account` for the migration (a documented
     scheme — e.g. `service = "alaya"`, `account` derived from the
     config key being migrated; pin the exact scheme in a code
     comment anchor).
  2. Pre-flight: confirm the host-platform keychain adapter is
     available **before mutating anything**. If not, fail with the
     remediation message (same wording family as `doctor` from slice
     2) and exit non-zero — no partial state.
  3. Prompt the operator for the secret value; mask it in any echo /
     log.
  4. Write the secret into the keychain (`writeKeychain` via the
     adapter).
  5. Verify the just-written entry resolves (`resolveSecretRef` on
     the new `keychain:` ref) — fail loudly if it does not (a stale
     keychain entry is acceptable residue; a config pointing at a
     keychain entry that was never written is not).
  6. Only then rewrite the runtime config `secret_ref` to
     `keychain:<service>:<account>`. The previous `env:` / `file:`
     ref's bytes on disk are unchanged (the env file / prior config
     value is left intact); the rewrite is confined to the
     `secret_ref` line(s) the migration owns.
  - Ordering rule (state it in a code comment anchor): keychain write
    → resolve-verify → config patch. A failure after the keychain
    write but before the config patch leaves an orphan keychain
    entry (acceptable, document it); the reverse (config points at
    `keychain:` but the entry is absent) must not happen.
- **Change**: define `--keychain --non-interactive` behaviour
  explicitly — pick one and pin it in this card and a code comment:
  (a) error ("`--keychain` requires interactive input"), or (b) the
  secret value comes from a documented env var (e.g.
  `ALAYA_KEYCHAIN_SECRET_VALUE`) consumed once and not persisted, or
  (c) it comes from the existing `--non-interactive` JSON config
  blob. Recommendation: (a) unless there is a concrete CI need.

### apps/core-daemon/src/secrets/keychain/index.ts (+ per-platform write impls)

- **Change**: if slice 1 did not already add it, add a
  `writeKeychain(service, account, value)` companion to the adapter
  interface and per-platform implementations:
  - macOS: `security add-generic-password -s <svc> -a <acct> -w <value> -U`
    (the `-U` updates an existing item).
  - Linux: `secret-tool store --label="alaya" service <svc> account <acct>`
    (value on stdin).
  - Windows: the write counterpart of slice 1's decided read
    mechanism.
  - All via the same injected subprocess runner so the write
    commands are unit-testable.

### apps/core-daemon/src/index.ts

- **Change**: wire the `--keychain` (and any non-interactive helper)
  argument into the install command construction.

### apps/core-daemon/src/__tests__/<install test file>

- **Change**: add unit cases with a stub keychain writer + a temp
  config dir — prompt consumed and masked; writer called with the
  derived `service` / `account`; config `secret_ref` rewritten;
  pre-flight failure (adapter missing) exits non-zero **before** any
  config mutation; `--non-interactive` behaviour matches the decided
  rule.

### apps/core-daemon/src/__tests__/install-keychain-migration.e2e.test.ts (new)

- **Change**: end-to-end migration test (this is the integration
  evidence for the operator-facing flow) — fresh temp config dir,
  starting from an `env:` (and a `file:`) ref, stubbed keychain
  read/write, scripted prompt input:
  - the prompt is consumed; the writer is called with the right
    `service` / `account` and the supplied value;
  - the config `secret_ref` is now `keychain:<svc>:<acct>` and the
    config still parses against `RuntimeGardenComputeConfigSchema`;
  - **the previous `env:` / `file:` ref's bytes on disk are
    unchanged** (assert the env file / prior config region
    byte-for-byte);
  - `resolveSecretRef` on the new config resolves via the stubbed
    keychain reader;
  - `alaya doctor` on the migrated config reports the keychain ref OK
    (uses slice 2's check).

### docs/v0.3/v0.3.0/reports/v0.3.0-slice-3.md (new)

- **Change**: completion report (incl. the manual-test transcript
  pointers — the real Linux/macOS/Windows `install --keychain`

## 3. Deferred

- OS-level secret rotation policies — out of scope for `#BL-009`;
  not re-deferred to a backlog issue (explicitly excluded by
  `docs/v0.3/README.md`).
- Bulk migration of historical config snapshots — out of scope; only
  the current runtime config is rewritten. Not re-deferred.
- `alaya install --keychain` for `secret_ref`s other than the
  Garden-compute provider ref (e.g. an embedding-provider key, which
  goes through a looser schema) — out of scope unless slice 1's
  schema work already covers it; if a real need surfaces, open a
  backlog issue at that point.

tool, point a temp runtime config's `garden_compute.secret_ref` at

`keychain:` ref while the old `env:` / `file:` ref is still in the
file, and `alaya doctor` is green.

## 6. Shared File Hazards & Dependencies

- `apps/core-daemon/src/cli/install.ts` is large (~500+ lines); the
  diff is confined to the `--keychain` branch — do not refactor
  surrounding code.
- `apps/core-daemon/src/index.ts` is touched by slices 1–3 for arg /
  dependency wiring — land in order, rebase forward.
- This card's review-loop is the strictest in the wave: the
  reversibility (old ref provably untouched) and atomicity (no
  half-migrated state, defined ordering) properties are the highest
  single risk in `#BL-009`. The E2E byte-equality assertion (AC4)
  and the pre-flight-before-mutation assertion (AC6) are mandatory.
- This card does not touch `docs/handbook/backlog.md` /
  `runtime-status.md`; slice 6 closes `#BL-009`.
