## Blocking

### B1 — `install --keychain` silently rewrites Garden `provider_kind`/`enabled` and bypasses the EventLog audit

`apps/core-daemon/src/cli/install.ts:752-773` (`patchPersistedGardenSecretRefIfPresent`),
mutation lands via `SqliteConfigRepo.set` →
`packages/storage/src/repos/config-repo.ts:49-55` (raw UPSERT).

The slice-3 card (`v0.3.0-slice-3-keychain-install.md` §2, step 6) says
the persisted-config rewrite "is confined to the `secret_ref` line(s)
the migration owns." The implementation instead writes:

```ts
const after = RuntimeGardenComputeConfigSchema.parse({
  ...parsedBefore,
  provider_kind: "official_api",
  enabled: true,
  secret_ref: secretRef
});
configRepo.set(RUNTIME_GARDEN_COMPUTE_CONFIG_KEY, after);
```

So an operator whose persisted `runtime:garden-compute` row had
`provider_kind: "host_worker"` (the host-worker garden loop) or
`"local_heuristics"`, and/or `enabled: false`, runs `alaya install
--keychain` to stash a secret and finds Garden silently switched to
enabled official-API mode — no preview, no confirm. Two problems:

1. **Scope.** Two governance fields the card did not authorize get
   rewritten. `install --keychain`'s documented goal is "rewrite the
   runtime config `secret_ref` to `keychain:<service>:<account>`",
   nothing more.
2. **Audit.** `configRepo.set()` is a raw UPSERT — it does *not* go
   through `EventPublisher.appendManyWithMutation`. The audited
   Garden-compute mutation path (`config-service.ts` `patchRuntimeGardenComputeConfig`)
   always emits an EventLog event; this install-side mutation emits
   none. The file-based install audit JSON only records `partial_state`
   *paths* (`install.ts:292-299`), so the DB-row change isn't recorded
   there either. This breaks the invariant "Governance, configuration,
   import/export, backup, and session trust changes are auditable."

Severity note: the red-team lens rated this Blocking; the correctness
lens flagged the same behaviour as an open question ("possibly
surprising — confirm intended"); the spec lens did not flag it. Merged
call: **Blocking** — an unaudited governance mutation that exceeds the
documented scope, with no test pinning the preserve-prior-state case, is
fix-before-close.

**Fix.** (1) On the success path, patch only `secret_ref` on the
persisted row — drop the `provider_kind` / `enabled` writes. If the
migration genuinely needs to enable official-API mode, prompt + confirm
and say so. (2) Either route the DB mutation through the audited
config-service path so it emits an EventLog event, or — at minimum —
record the config key + before/after `provider_kind`/`enabled`/`secret_ref`
in the install audit JSON payload so the change is auditable. (3) Add a
test asserting a pre-existing `provider_kind: "host_worker"` survives
`install --keychain`. The rollback path (`restorePersistedGardenConfig`,
`install.ts:775-784`) already restores the full prior config — only the
success path is out of scope.

## Important

### I1 — keychain subprocess runner has no timeout; a locked backend can hang daemon boot and `alaya doctor`

`apps/core-daemon/src/secrets/keychain/index.ts:123-141`
(`defaultKeychainSubprocessRunner`) passes `timeout: options.timeoutMs`
to `spawnSync`, but no caller ever sets `timeoutMs` — `spawnSync` with
`timeout: undefined` blocks forever. This runner is on:

- daemon startup — `index.ts:582-589` → `buildGardenComputeRoutingProviders`
  → `canResolveOfficialGardenProvider` (`index.ts:906-921`) →
  `resolveGardenSecretRefValue` → `readPlatformKeychainSecret`, and again
  on every `patchRuntimeGardenComputeConfig`. A locked `secret-tool`
  collection on a headless box, or a macOS Keychain that pops the
  "security wants to use your confidential information" dialog, makes
  the daemon never finish booting when a `keychain:` Garden ref is
  configured.
- `alaya doctor` / status — `cli/register.ts:141,156` resolves the
  secret ref (twice — see N1). A stuck backend hangs the one command an
  operator runs when something is wrong.
- `garden-credential.ts:132` and the `install --keychain` verification
  step — same.

**Fix.** Have the default runner (or every caller) pass a bounded
`timeoutMs` (5–10 s) and map a timed-out `spawnSync`
(`result.error?.code === "ETIMEDOUT"` / `result.signal === "SIGTERM"`)
to `keychain_tooling_unavailable` (or a new `keychain_locked` kind) with
a remediation pointing at unlocking the keychain. Longer-term, the
boot-path secret resolution should not be synchronous at all, but a
bounded `spawnSync` closes the liveness hole now.

### I2 — `@do-soul/alaya-protocol` EventLog surface moved but the workspace is still `0.2.0` on `main`

`packages/protocol/src/events/garden.ts:82` widens `secret_ref_kind` to
`z.enum(["env","file","keychain"])`; the `semver-surface` snapshot moved
one hash line; `packages/protocol/package.json` and
`apps/core-daemon/package.json` are still `"version": "0.2.0"`. Per
invariant §25, that is the state the contract explicitly forbids: a
strict consumer parsing with the old `z.enum(["env","file"])` rejects a
`secret_ref_kind: "keychain"` payload, and a `^0.2.0` pin would receive
the new surface under the old version.

This is *acknowledged* — the commit message marks the bump "not done …
slice 6", and slice 6 is intentionally unexecuted. So it is a known
checkpoint state, not an oversight. Recorded here because (a) the review
must note `main`'s head is currently §25-inconsistent, and (b) slice 6's
version bump + release notes + maintenance §25 note are a hard
prerequisite before this surface ships. **Do not publish
`@do-soul/alaya-protocol` from any commit between `d26998e` and the
slice-6 bump.**

### I3 — `install --keychain` parsing path (`installArgsSchema`) is untested; partial-failure leaves an orphan keychain entry with no audit trail of it

Two related coverage gaps:

- `apps/core-daemon/src/cli/install.ts:411-475` — `installArgsSchema.safeParse`
  gained non-trivial `--keychain` token handling (re-ordering splices,
  the `keychain && tokens.length <= 1` short-circuit, the interactive
  case). No test exercises `safeParse`: `cli-install.test.ts` only adds
  `force`/`keychain` fields to existing `handler(...)` calls, and the
  e2e test calls `command.handler` directly with `{keychain:true}`,
  bypassing the schema. Edge this hides: `install --keychain
  --non-interactive garbage` silently discards `garbage` and then fails
  with "requires interactive input" instead of an unexpected-argument
  error. **Fix.** Add a unit test over `installArgsSchema().safeParse([...])`
  for `["--keychain"]`, `["--keychain","--force"]`, `["--keychain","extra"]`
  (must reject), `["--keychain","--non-interactive"]`.
- `apps/core-daemon/src/cli/install.ts:264-289` — if `writeKeychain`
  succeeds and any later step throws (verification, the `.env` atomic
  write, `patchPersistedGardenSecretRefIfPresent`), the `catch` rolls
  back the `.env` file and the persisted config and writes a `"failed"`
  audit row — but the keychain entry is **not deleted and not listed in
  `partialState`**, so the `"failed"` audit JSON shows a clean-looking
  `partial_state` with no mention of the orphaned secret. The card calls
  a stale keychain entry "acceptable residue" but says to *document it*;
  the audit record is the natural place. Across repeated failed installs
  these accumulate, and the failed-audit guard then blocks the next
  `install` unless `--force` (which also doesn't clean it). **Fix.**
  Include `keychain:<service>:<account>` in the `"failed"` install
  audit's `partial_state` / a dedicated `keychain_orphan` field whenever
  `writeKeychain` returned ok but the install later failed, with a
  one-line remediation (`security delete-generic-password` /
  `secret-tool clear` / PowerShell `PasswordVault.Remove`). Best-effort
  deletion on rollback is optional.

## Nice-to-have

### N1 — `alaya doctor` resolves a keychain ref twice
`apps/core-daemon/src/cli/register.ts:156` (`resolveGardenRoutingDecision`)
and `:179` (`keychainCheckField`) each call
`resolveSecretRef(config.secret_ref)` — two `spawnSync` invocations per
`alaya doctor` for `keychain:` refs. Resolve once, thread the result.

### N2 — `keychain:<service>:<account>` parsing is duplicated in four sites with divergent error text
`apps/core-daemon/src/secrets.ts:174-186` (`parseKeychainRef`),
`apps/core-daemon/src/cli/register.ts:201-208`,
`apps/core-daemon/src/services/env-file-service.ts:423-429`,
`packages/protocol/src/app-config.ts:84-89` — same grammar ("strip
`keychain:`, split on `:`, require two non-empty segments"), three
idioms, three error messages. The secret-ref grammar is a contract
surface (protocol validates, `secrets.ts` resolves, `env-file-service`
re-validates on write, `register.ts` re-parses for doctor); drift here
produces inconsistent accept/reject between layers. **Fix.** Put one
`parseSecretRefKeychainTarget(ref)` (or a `SECRET_REF_KEYCHAIN_PATTERN`
regex) in `@do-soul/alaya-protocol` (the zod-only leaf already owns the
`secret_ref` superRefine — dep direction stays legal) and import it from
the three core sites. Fold the duplicated `secretRefKind(ref)` between
`config-service.ts:407-419` and `garden-credential.ts:105-118` into the
same helper.

### N3 — `keychain:` ref segments accept embedded/trailing whitespace
`apps/core-daemon/src/secrets.ts:172-190`, `packages/protocol/src/app-config.ts:85-90`,
`apps/core-daemon/src/services/env-file-service.ts:426-431` — `"keychain:alaya: openai"`
and `"keychain:alaya:openai "` pass (segment is non-empty after
`split(":")`) and the whitespace is forwarded verbatim to `security -a`
/ `secret-tool account`. Tighten the segment rule (reject if `segment
!== segment.trim()`, or restrict to `[A-Za-z0-9._-]+`) so the config
surface and the resolver agree on a canonical shape. On Windows the
service/account become `$args[0]`/`$args[1]` after `powershell.exe
-Command <script>` (`secrets/keychain/windows.ts:35-44`); the same
charset restriction also closes the Node-Windows command-line quoting
edge cases there. Low risk (a malicious `alaya.env` is the only way to
inject odd values, and that attacker already controls the env file), but
worth the cheap hardening.

### N4 — `install.ts` is growing as a god-file (~860 lines after +378); the masked-stdin reader uses an unsafe `as`-cast
`apps/core-daemon/src/cli/install.ts` now carries the legacy install
flow, `executeKeychainInstall`, `readSecretLine` / `readMaskedTtySecretLine`
(~110 lines of raw-mode TTY masking), `patchEnvWithGardenKeychainRef`,
`patchPersistedGardenSecretRefIfPresent` / `restorePersistedGardenConfig`,
audit/rollback bookkeeping, two `format*Error` switches, and the args
parser. **Fix (mechanical, e2e already pins behaviour).** Extract
`install/keychain-install.ts` (the keychain flow + its `patch*` /
`restore*` / `format*` helpers) and `install/masked-stdin.ts`
(`readSecretLine` + `readMaskedTtySecretLine`); `install.ts` keeps the
command spec, args parsing, and the legacy flow. Separately,
`install.ts:679-697` does `const input = stdin as RawModeReadable;` onto
a hand-rolled all-optional interface, then `?.`-calls every method —
replace with the real `node:tty` `ReadStream` type (the codebase already
threads `ctx.isTTY`), or accept `NodeJS.ReadStream` in the signature.

### N5 — macOS `security -i` quoting for adversarial secret values is unverified; add a comment and a test
`apps/core-daemon/src/secrets/keychain/macos.ts:41-43,74-78`
(`quoteSecurityInteractiveArg`) — the secret-on-stdin `security -i` form
single-quotes each field and escapes `'` as `'\''`. The shape is right
(the secret is off argv; `keychain-adapters.test.ts:57` asserts that),
but the only tested value is the literal `"secret"`. The assumption that
`security`'s line tokenizer follows POSIX-shell single-quote semantics
(`\` literal inside `'...'`) is unproven, and the commit itself flags
the missing macOS transcript. If the tokenizer treats `\` as an escape
inside single quotes, a backslash in a secret is silently corrupted (a
wrong key stored — not a leak). **Fix.** Add unit cases for secrets
containing `'`, `\`, a leading `-`, `$(...)`; capture the real macOS
round-trip transcript the commit acknowledges is missing; add a
one-line comment on `writeMacosKeychainSecret` ("argv `-w <secret>`
form leaks the secret to `ps`; the interactive `-i` stream keeps it off
argv"). `readSecretLine` already strips `\r`/`\n`, so newline-in-secret
isn't reachable from `install --keychain` — note that too. (A control-
char guard on the entered secret before write is the broader fix; see
the macOS truncation note in the correctness draft.)

### N6 — smaller hardening / honesty items
- **Windows: `PasswordVault` load failure misreported as "entry not
  found".** `secrets/keychain/windows.ts:30-65` — if PowerShell is
  present but the `Windows.Security.Credentials.PasswordVault` WinRT
  type fails to load (some Server SKUs), the non-zero exit maps to
  `keychain_entry_not_found`; the remediation then tells the operator to
  populate an entry that can never be read. Distinguish the WinRT-load
  failure via stderr if feasible; otherwise note the limitation in the
  script comment.
- **Windows: machine-wide PowerShell transcription captures the
  read-back secret.** `secrets/keychain/windows.ts:8-17` reads via
  `[Console]::Out.Write($credential.Password)`; if Group Policy enables
  `Start-Transcript` system-wide, console output (the secret) lands in
  the transcript directory. `-NoProfile` doesn't disable policy-driven
  transcription. Exotic — document it; prefer a non-console channel for
  the secret if a clean one exists.
- **No cross-process install lock.** `install.ts:222-234,253-299` —
  `detectBlockingPriorAudit` runs, then `install --keychain` blocks on
  the interactive prompt, then writes the `"started"` audit; two
  concurrent invocations both pass the guard and last-writer-wins on the
  `.env` / DB row. Low risk for a single-operator CLI, but the
  audit-first ordering doesn't actually serialize installs. The project
  already has a cross-process lock for `applyRuntimeEmbeddingConfigFiles`;
  reuse it around the install body.
- **`keychain:` refs aren't pinned to an Alaya-owned namespace.**
  `secrets.ts:142-170` — a `keychain:` ref can point at any service/
  account in the current user's keychain. Bounded (same-user keychain;
  an env-file attacker could already set `env:THEIR_VAR`), so not an
  escalation — but consider pinning resolvable services to an `alaya*`
  prefix unless an explicit allow-list says otherwise, and `doctor`
  already surfaces the resolved service/account for the configured ref.
- **`readPlatformKeychainSecret`'s `{ platform? }` override seam is
  dead** in both prod and tests (`secrets/keychain/index.ts:38-46` and
  siblings) — the keychain-adapters test imports the per-platform
  functions directly with an injected runner. Either add one
  `readPlatformKeychainSecret("s","a",{platform:"win32",runner})`
  assertion or drop the `platform` field.
- **`RuntimeSecretRefSchema` superRefine changes are invisible to the
  semver snapshot** (`packages/protocol/src/app-config.ts:68-96`) —
  only field names are hashed, so adding `keychain:` (and a future
  *breaking* change like dropping `file:`) doesn't move the snapshot.
  Pre-existing limitation; backlog note: validator-shape changes on
  `secret_ref` are §25-relevant but uncaught.
