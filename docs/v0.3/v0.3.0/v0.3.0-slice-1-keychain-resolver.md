## 1. Background & Goal

**Background.** `apps/core-daemon/src/secrets.ts` `resolveSecretRef`
currently dispatches only on the `env:` and `file:` prefixes;
`ResolvedSecret.origin` is the union `"env" | "file"`,
`ResolveSecretError` has five kinds, and the module stays testable by
injecting a `SecretRefReader` (`readEnv` / `readFile`). The runtime
config schema enforces the same two prefixes:
`RuntimeSecretRefSchema` in `packages/protocol/src/app-config.ts` is
a `z.string().superRefine(...)` that rejects anything not
`env:<ident>` or `file:/abs`. `secret_ref_kind` in
`packages/protocol/src/events/garden.ts` is `z.enum(["env",
"file"])` on a Garden EventLog payload. `#BL-009` asks for a third
adapter — the OS keychain — so an operator does not have to keep an
API key in a plaintext config file or environment variable.

Both `RuntimeSecretRefSchema` (config surface) and `secret_ref_kind`
(EventLog payload surface) are §25-covered. Adding `keychain:` to
the accepted shapes and `keychain` to the enum is an additive change
on two covered surfaces — a §25 **minor** bump — which is why this
release is v0.3.0, not v0.2.1, and why `semver-surface.test.ts` must
be re-snapshotted in this card.

**Goal.** Give `resolveSecretRef` a `keychain:<service>:<account>`
branch backed by three platform adapter modules and a `readKeychain`
reader hook, widen `ResolvedSecret.origin` / `ResolveSecretError` /
`RuntimeSecretRefSchema` / `secret_ref_kind` for the new shape,
re-snapshot the §25 surface, and declare the SemVer step in this
card.

## 2. Allowed Scope

### apps/core-daemon/src/secrets.ts

- **Change**: add a `keychain:` prefix branch to `resolveSecretRef`.
  The ref shape is `keychain:<service>:<account>` — exactly two
  non-empty, colon-free segments after the prefix; anything else is
  `{ kind: "malformed", ref, reason }`. (Pick a parser that does not
  silently accept extra colons; document the chosen rule in a code
  comment that is an invariant anchor, not narrative.)
- **Change**: add `readKeychain(service, account)` to the
  `SecretRefReader` interface as a **non-optional** method (so the
  type checker finds every construction site). The default reader
  routes to the host-platform adapter (see the `keychain/` modules).
- **Change**: widen `ResolvedSecret.origin` to `"env" | "file" |
  "keychain"`; on success the keychain branch returns `{ ref, value,
  origin: "keychain" }`. An empty stored value returns `{ kind:
  "empty", ref, origin: "keychain" }` (mirror the env/file behaviour;
  trim consistently).
- **Change**: add `ResolveSecretError` kinds for the keychain
  failure modes — at minimum a "platform keychain tooling not
  available" kind (the adapter binary is missing / the platform has
  no adapter) and a "keychain entry not found" kind. Both carry a
  `reason` suitable for a `doctor` remediation message and the
  `service` / `account` (never the secret value). Reuse `malformed`
  for shape errors.

### apps/core-daemon/src/secrets/keychain/index.ts (new)

- **Change**: export the host-platform dispatch (`process.platform`
  → macOS / Linux / Windows adapter, or a clean "no adapter for this
  platform" error), the shared adapter interface, and the shared
  error/result types. All adapters take an **injected subprocess
  runner** (a small `(cmd, args, opts) => { code, stdout, stderr }`
  shape) so command construction is unit-testable on any OS and the
  real spawn is the only thing that differs at runtime.

### apps/core-daemon/src/secrets/keychain/macos.ts (new)

- **Change**: implement the macOS adapter via
  `security find-generic-password -s <service> -a <account> -w`.
  Distinguish "binary not on PATH" (→ tooling-not-available kind)
  from "item not found" (`security` exit status / stderr → entry-not-found
  kind) from a successful single-line secret read. Never fall through
  to a silent success.

### apps/core-daemon/src/secrets/keychain/linux.ts (new)

- **Change**: implement the Linux adapter via
  `secret-tool lookup service <service> account <account>` (libsecret).
  "binary not on PATH" → tooling-not-available kind with a
  remediation naming the `libsecret-tools` package; non-zero exit /
  empty output → entry-not-found kind; success → the printed secret.

### apps/core-daemon/src/secrets/keychain/windows.ts (new)

- **Change**: implement the Windows read path. **Decide the concrete
  mechanism before freezing this card** (run `do-it-grill` on it):
  `cmdkey` can write/list but cannot read a stored secret back, so
  the read path is most likely PowerShell over
  `[Windows.Security.Credentials.PasswordVault]` or a `CredRead` /
  `CredEnumerate` invocation. Document the chosen mechanism and the
  prerequisite credential-manager state in a code comment anchor and
  in this card's §2 once decided. Same three outcomes
  (tooling-not-available / entry-not-found / success).

### apps/core-daemon/src/__tests__/secrets.test.ts

- **Change**: add `keychain:` cases against a stub `readKeychain`:
  resolved → `origin: "keychain"`; malformed shapes (`keychain:`,
  `keychain:onlyservice`, `keychain:a:b:c`, empty segments) →
  `kind: "malformed"`; tooling-not-available → that kind with a
  remediation `reason`; entry-not-found → that kind; empty stored
  value → `kind: "empty", origin: "keychain"`.

### apps/core-daemon/src/__tests__/keychain-adapters.test.ts (new)

- **Change**: with a stub subprocess runner, assert each platform
  adapter constructs the documented argv (`security find-generic-password
  -s … -a … -w`, `secret-tool lookup service … account …`, the
  decided Windows command) and maps the three outcome classes
  correctly. Runs on any OS (the runner is stubbed).

### packages/protocol/src/app-config.ts

- **Change**: extend `RuntimeSecretRefSchema` to also accept
  `keychain:<service>:<account>` (same two-non-empty-segments rule
  as the resolver). Everything else (`env:` identifier rule, `file:`
  absolute-path rule, rejection of unknown schemes) is unchanged.

### packages/protocol/src/events/garden.ts

- **Change**: extend `secret_ref_kind` from `z.enum(["env", "file"])`
  to `z.enum(["env", "file", "keychain"])` (additive enum value, not
  a discriminator key).

### packages/protocol/src/__tests__/{app-config,garden-events}.test.ts

- **Change**: add cases — `RuntimeSecretRefSchema` accepts
  `keychain:svc:acct` and still rejects garbage; `secret_ref_kind`
  parses `"keychain"`.

### packages/protocol/src/__tests__/semver-surface.test.ts + __snapshots__/semver-surface.test.ts.snap

- **Change**: regenerate the committed snapshot. The `secret_ref`

  `apps/core-daemon/src/index.ts`, `apps/core-daemon/src/garden/credential.ts`,

## 6. Shared File Hazards & Dependencies

- `packages/protocol/src/__tests__/__snapshots__/semver-surface.test.ts.snap`
  will move the moment `RuntimeSecretRefSchema` or `secret_ref_kind`
  changes — that is the point of the test. The **only** correct fix
  is updating the snapshot in this card *and* bumping the workspace
  version to `0.3.0` in v0.3.0-slice-6. Updating the snapshot without
  the version bump is exactly the failure §25 guards against.
- `SecretRefReader` is widened here (non-optional `readKeychain`);
  all construction sites are swept in this card (see §2). The type
  is workspace-internal (no MCP/EventLog/config surface), so this is
  out of §25 scope per the §25 carve-out.
- The Windows read mechanism is under-specified by `#BL-009` as
  written (`cmdkey` cannot read secrets back); resolve it with
  `do-it-grill` before freezing this card's §2.
- `packages/protocol/src/app-config.ts` and `events/garden.ts` are
  §25-covered files — this card cites §25 and declares the SemVer
  step (minor). `docs/handbook/invariants.md` itself is **not**
  edited here; v0.3.0-slice-6 adds the §25 worked-example note.
