## Production Evidence (inherited from v0.3.3 dogfood pass)

The v0.3.3 dogfood pass on the maintainer's WSL2 host (recorded
in `.do-it/findings/v0.3.3-dogfood-evidence.md`) established the
following end-to-end facts that v0.3.4 inherits unchanged:

- **Q1 — RECALLS edges persisted.** All 6 rows of
  `memory_graph_edges` are `edge_type = "recalls"`, produced from
  a single used recall report at `2026-05-14T01:03:36.815-819Z`
  (3 candidate memories interconnected, 6 directed edges in a 5ms
  fanout window).
- **Q2 — used → edge_created → MCP audit chain.** EventLog rows
  in a 30ms window: `memory.usage_reported` (internal) → 6 × `soul.graph.edge_created`
  → `soul.context_usage.reported` (MCP audit). Sequence
  conforms to invariants §7 / §10 (EventLog → DB → audit before
  broadcast).
- **Q3 — graph_support enters recall scoring from persisted
  edges.** The two memories that appear most frequently in the
  RECALLS edge endpoints (`666b3995…` / `4d6f3950…`, each at the
  endpoint of 4 of the 6 edges) carry `score_factors.graph_support
  = 0.2` in subsequent `soul.recall` responses; memories outside
  the edge set carry 0.
- **Q4 — cold reallocation path observed at code level.**
  `resolved_activation_weights` is present on every result;
  natural recall traffic during the dogfood window landed in a
  mixed state, which means the cold reallocation trigger was not
  exercised from production data. The v0.3.4 mixed-cold test
  (Slice 3) closes this gap at the unit-test boundary.
- **Q5 — doctor graph_health is truthful.** `alaya doctor` prints
  `graph health: healthy memory_edges=6 path_relations=1
  latest_path_event=none`, numerically consistent with the
  raw SQL.
