# v0.3.9 Bench Pre/Post Diff

This document carries the bench-evidence half of the v0.3.9 truth-up:
what changed in the bench harness itself, what got re-archived, and
how the post-fix KPIs compare to the pre-fix Pass A baselines.

It is the source the release notes, closeout, and Inspector trend
panel link to when they cite "after the v0.3.9 fix the bench reports …".
Do not duplicate the prose here in those files; link to the section.

## 1. What changed in the bench harness itself

| Area | Pre-v0.3.9 (Pass A) | v0.3.9 (Pass B) | Reason |
|---|---|---|---|
| `apps/bench-runner/src/locomo/runner.ts` `evaluated_count` denominator | `window.length` (count of conversations evaluated; ≤ 10) | `totalQa` (count of scoreable QAs actually run; ≈ 200 per conversation) | The LoCoMo bench scores per-QA hits; reporting a denominator in conversation units made the per-QA R@K stats appear over a tiny sample. `evaluated_count` now matches the scoring unit, and the `evaluated_count <= sample_size` refinement holds in QA units. |
| `apps/bench-runner/src/locomo/runner.ts` `sample_size` semantic | `conversations.length` (≤ 10) | `resolveLocomoSampleSize(conversations)` — sum of QAs that carry non-empty `evidence` across the full dataset | Mirrors the runner's actual scoring contract: LoCoMo category-5 adversarial entries omit evidence by design and are excluded from the denominator both at sample_size and at evaluated_count. |
| `packages/eval/src/wilson-ci.ts` `SampleSizeLabel` cascade | 3 tiers: `smoke` (<100) / `shard_merged` (latency=`worst_shard_bound`) / `full` (≥100) | 4 tiers: `smoke` (≤50) / `staged` (51-200) / `shard_merged` (201-499 OR latency=`worst_shard_bound`) / `full` (≥500) | A `100/500` partial run was previously labelled `full`, which inflated the trust signal carried into release notes. The new cascade names the four release-confidence buckets — see §3 below for what each label promises. |
| `packages/eval/src/cohort-attribution.ts` (new) | absent | `computePlaneAttribution()` + `extractPlaneAttributionRows()` + `shareOfPlane()` | Per-plane admission-share helper. Reads `plane_winning_admission` off the diagnostics sidecar; flags single-plane domination (e.g. `session_surface_cohort > 50%` is a recall-stack collapse signal). Test in `packages/eval/src/__tests__/cohort-attribution.test.ts`. |
| `docs/bench-history/public/latest-baseline-embedding-on.json` (new) | absent | sibling pointer to the most recent non-FAIL embedding-on archive | Decouples the embedding-on regression chain from the embedding-off chain. An embedding-on FAIL no longer pollutes the disabled-embedding baseline. |
| `docs/bench-history/public-pre-v0.3.9/` (new directory) | n/a | frozen Pass A archives | The pre-fix archives carry the old denominator / label semantics; freezing them under a sibling directory preserves the historical record without polluting the post-fix diff chain. |

## 3. Sample-size label cascade — what each label promises

| Label          | Bucket                                                            | Release claim it underwrites |
|---             |---                                                                |--- |
| `smoke`        | `evaluated_count ≤ 50`                                            | Tripwire only — verifies the code path runs end-to-end on a small slice. Never quoted as a quality claim. |
| `staged`       | `51 ≤ evaluated_count ≤ 200`                                      | Staged-release confidence: enough data to gate a hotfix or staged rollout, but the 95% Wilson CI widens above the raw 2pp / 5pp regression bands (`ciAwareBand` widens automatically). |
| `shard_merged` | `201 ≤ evaluated_count ≤ 499` OR `latency_source = worst_shard_bound` | Cross-shard merged: latency reads as the upper bound across N shards (not an exact percentile). KPIs are real but latency comparisons are bounded above; release-notes should cite the bench-history `report.md` `latency_source` field. |
| `full`         | `evaluated_count ≥ 500`                                           | Full dataset, exact percentiles, release-grade evidence. The only label that may be quoted as "Alaya recall at production scale". |

`latency_source = worst_shard_bound` always wins the label regardless
of `evaluated_count` because the latency channel is structurally
upper-bound, not a true run-wide percentile.

### Recall sensitivity to dimension diversification (load-bearing observation)

The pre-rotation baseline at `2026-05-16T021654Z-0410c0d` reported
R@5 = 77% on LongMemEval-S 100. The post-rotation baseline at
`2026-05-17T065105Z-75d418c` reports R@5 = 1.0% on the same dataset
and the same number of evaluated questions, but R@10 = 63% — the
hits are present in the recall pool, they are no longer in the
top-5.

The rotation changed seeded `object_kind` from a uniform
`"fact"` distribution to a 5-slot rotation
(`fact / preference / decision / constraint / outcome`). Through
`toMemoryDimension`, that flips the seeded `memory_entry.dimension`
from a uniform `FACT` to a four-dimension mix (`FACT` / `PREFERENCE` /
`DECISION` / `CONSTRAINT`). All 100 returned candidates land in the
`warm` storage tier; the `degradation_reasons` are empty (no
cascade engagement). The drop is therefore explainable as a
recall-ranking effect — Alaya's scoring favours `FACT` rows over
mixed-dimension rows in the top-5 slots, even though the underlying
`memory_entry` rows are identical between branches.

This is a real finding the v0.3.9 producer-side diversification
surfaces. The previous 77% number was an artefact of a uniform-
`FACT` workspace shape that the live system would never see in a
real long-running deployment (where Garden routes by object_kind
into all four dimensions). The new R@5 is the honest "Alaya recall
under realistic dimension mix" measurement. Tuning the recall
scoring to balance across dimensions is a follow-up — tracked in
the closeout's L0+ list as "recall scoring sensitivity to memory
dimension surfaced by L1 producer-side diversification".

### Embedding plane-admission impact (load-bearing observation)

Read off the diagnostics sidecar at
`docs/bench-history/public/2026-05-17T104037Z-294070f/longmemeval-diagnostics.json`
via `scripts/analyze-recall-bias.mjs`. The 100-question embedding-on
run carries 170 gold entries vs the 1000 entries in the embedding-off
sibling — the recall service emits fewer-but-tighter candidates when
the embedding plane is online, so the per-question gold pool shrinks.

Final-rank distribution among delivered gold:

|              | embedding-off (n=1000 delivered) | embedding-on (n=71 delivered) |
|---           |---                               |---                            |
| `rank = 1`   | 0.0%                             | 19.7%                         |
| `rank 2-5`   | 1.5%                             |  5.6%                         |
| `rank 6-10`  | 98.5%                            | 74.6%                         |
| `rank > 10`  | 0.0%                             |  0.0%                         |

Plane that admitted the delivered gold:

|                            | embedding-off | embedding-on |
|---                         |---            |---           |
| `lexical`                  | 52.9%         | 49.3%        |
| `temporal_proximity`       | 38.2%         | 33.8%        |
| `evidence_anchor`          |  4.4%         |  4.2%        |
| `session_surface_cohort`   |  4.4%         |  2.8%        |
| `null`                     |  0.0%         |  9.9%        |

The embedding plane (admissions show up as `null`-tagged here because
the recall service does not currently tag the embedding plane name —
see closeout carry-forward #22) rescues ~10 hits per 100 questions
into `rank = 1`, but `temporal_proximity` still wins ~1/3 of all
admissions and remains the dominant filler of slots 6-10. The R@5
recovery from 1% to 17% is therefore narrow: it is the top-of-list
boost from semantic similarity, not a structural fix to the plane
admission bias. Tracked in closeout carry-forward #23 as
"recall plane admission concentration".
