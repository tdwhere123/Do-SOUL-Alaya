# v0.3.8 Closeout Report

## Status

Implementation checkpoint on branch `main` on 2026-05-16. v0.3.8 lands
seven layers of ontology mid-layer work that were previously staged as
"v0.4 future work" in protocol comments. Bench evidence on three
LongMemEval-S surfaces shows R@5 lifts on every surface, with latency
unchanged or improved.

## Scope Compliance

- MCP tool names, request schemas, and response schemas are unchanged.
  The single schema-level change is additive: `SoulOpenPointerContentSchema`
  gains optional `gist` / `excerpt` fields so EvidenceCapsule pointer
  resolution returns its raw turn material; the fields are absent on
  the MemoryEntry path.
- Protocol zod schemas (`memory-graph.ts`, `path-relation.ts`,
  `evidence-capsule.ts`) are unchanged.
- One new SQL migration (`068-evidence-capsule-fts.sql`) creates the
  evidence-side FTS5 virtual table + insert/update/delete triggers.
  No ontology table migrations.
- No runtime config schema change.

## Delivered

### Layer 1 — Object initialization

- `INITIAL_ACTIVATION_FROM_CONFIDENCE_FACTOR` bumped 0.5 → 0.6 so the
  scoring base weight has a stronger initial signal on freshly
  materialized memories. Verified rank1-rank10 score gap on
  disabled-100 lifted ~0.02 → ~0.16.
- `createContentPreview` drops the `originPlane === "global"`
  requirement. Workspace-local memories with `manifestation ===
  "full_eligible"` now serve their full content.

### Layer 2 — EvidenceCapsule + distillation

- `materialization-router.buildDistilledFact` is the new producer of
  `MemoryEntry.content`. Caller may supply
  `signal.raw_payload.distilled_fact`; otherwise a rule-based
  fallback takes up to 2 sentences capped at 280 chars across both
  Latin (.!?;) and CJK (。！？；) terminators. Raw turn lives in
  `EvidenceCapsule.gist / .excerpt`.
- `soul.open_pointer` now resolves to `EvidenceCapsule` when the
  memory lookup misses, scoped per workspace via
  `EvidenceService.findByIdScoped`. Attached agents can dereference
  `evidence_refs[i]` back to raw turn material through the same MCP
  entry point used for memory pointers.
- Migration `068-evidence-capsule-fts.sql` adds an FTS5 virtual table
  + triggers mirroring the memory-FTS pattern from migration 019.
- Recall lexical plane consults evidence FTS in addition to memory
  FTS. Evidence-side hits are resolved back to owning memories via
  `MemoryEntry.evidence_refs` (JSON `LIKE ? ESCAPE '\\'` with
  `%"<id>"%` so non-UUID ids cannot fuzz-match).

### Layer 3 — Edge producers

- The four staged `MemoryGraphEdgeType` values
  (`supersedes` / `contradicts` / `exception_to` /
  `incompatible_with`) now have writers. Caller-explicit hints via
  `raw_payload.{supersedes,exception_to,contradicts,
  incompatible_with}_refs` produce the corresponding edges at
  materialization time. The `createSourceMemoryEdges` helper was
  generalized into `createEdgesFromRawPayloadRefs` so all four hint
  types share one fire-and-forget writer.
- New `ConflictDetectionService` in `packages/core` runs at
  materialization time. Rule shape: same dimension + same scope +
  tag Jaccard ≥ 0.5 + token Jaccard < 0.35 ⇒ contradicts;
  cross-dimension or cross-scope + tag Jaccard ≥ 0.5 ⇒
  incompatible_with. Token comparator uses Unicode-aware
  tokenization across Latin and CJK.
- Optional LLM port wired via env
  (`ALAYA_CONFLICT_LLM_PROVIDER_URL` / `_API_KEY` / `_MODEL` /
  `_TIMEOUT_MS`). When configured, an openai-compatible chat
  completion request classifies each borderline neighbor as
  `contradicts` / `incompatible_with` / `none`. Hook is bounded to
  `llmMaxPairsPerNewMemory` per materialization (default 4).
- The whole service is opt-in: `ALAYA_CONFLICT_DETECTION_ENABLED=1`
  switches it on. Default off — each materialization would otherwise
  pay O(workspace_size) for `findByDimension +
  findByWorkspaceId`, which a 100-question LongMemEval seeding burst
  measured as ~150× the no-detection p95. The env-gate restored bench
  latency to baseline.

### Layer 4 — PathRelation propose

- New `PathRelationProposalService` in `packages/core`. Counts
  in-process co-usage events per memory pair; on the
  `PATH_RELATION_PROPOSE_THRESHOLD = 3` co-usage of a pair, writes a
  PathRelation with `recalls_edge_co_usage` evidence basis, initial
  strength 0.3, `bidirectional_asymmetric` direction, and
  `recall_bias 0.5`. Existing relations between the same pair are
  detected via `findByAnchors` and skipped.
- `crossLinkRecalledMemories` in `mcp-memory-tool-handler.ts` calls
  `pathRelationProposalService.onCoUsage(usedObjectIds, workspaceId)`
  after writing RECALLS edges so PathRelation propose lives alongside
  the RECALLS-edge writer at one boundary.
- `PathPlasticityService` now has a real source of paths to evolve.
  Prior to this commit PathRelation was a half-dead schema with a
  writable surface but no producer.

### Layer 5 — Recall engineering

- `task-surface-builder` default budgets widened:
  chat 10→15 entries (2000→3000 tokens),
  analyze 25→30 (4000→5000),
  build 15→20 (3000→4000),
  govern 20→25 (3500→4500).
  Targets the `budget_dropped` miss class that dominated v0.3.7
  cross-question diagnostics.
- `session_surface_cohort` seed-cohort branch gains a pre-admission
  dominance guard: when the would-be cohort (per-seed ±radius union)
  covers more than half of the tier pool, the branch is skipped so
  evidence_anchor / domain_tag_cluster / lexical compete for delivery
  instead of being drowned by trivial admit-everything cohort fanout.
- `graph_expansion` and `path_expansion` seeds now pass a
  `usage_proof` gate: a seed only projects its neighbors when it is
  either a governance-attested winner (`winnerMemoryIds`) or carries
  at least one inbound RECALLS edge. Pure lexical-match seeds no
  longer pull graph / path neighbors as structural support.

## Bench Evidence

All three LongMemEval-S surfaces show real improvements after the v0.3.8
stack vs the v0.3.7 honest baseline (`6128977`):

| Surface | v0.3.7 honest | v0.3.8 | Δ R@5 | Δ latency p95 |
|---|---:|---:|---:|---:|
| disabled-100 | R@5 69.0% | R@5 77.0% | +8 pp | 161 → 90 ms |
| multi-turn 50×3 | R@5 84.0% | R@5 88.0% | +4 pp | 95 → 72 ms |
| cross-question 50 | R@5 62.0% | R@5 70.0% | +8 pp | 4065 → 2908 ms |

Sidecar evolution (disabled-100):
- Rank 1 vs rank 10 score gap: 0.04 → **0.16** (4× wider).
- `plane_winning_admission` rotated: `session_surface_cohort` 94% →
  `temporal_proximity` 87%; cohort plane retreats after the dominance
  guard kicks in.
- Miss class: `hit_at_5` 69 → 77, `under_ranked` 4 → 1,
  `budget_dropped` 21 → 16, `candidate_absent` 6 → 6.

Sidecar evolution (cross-question 50):
- `plane_winning_admission`: `session_surface_cohort` 41 (82%),
  `evidence_anchor` 7 (14%), `temporal_proximity` 2 (4%).
- Miss class: `budget_dropped` 13 → 9, `under_ranked` 5 → 5,
  `candidate_absent` 1 → 1.
- `first_half` 64% → 72%, `last_half` 60% → 68%. Both halves lift
  by +8 pp; the gap between halves remains, so cross-question
  accumulation does not yet measurably help later questions. This is
  consistent with PathRelation propose only firing at K=3 co-usage
  and LongMemEval questions being topically independent.

Disabled-500 full run: in progress; will land alongside the final
release notes commit. Phase 6 verification runs disabled-500 to
confirm the gain holds at full sample.

## Honest acknowledgements

- Cross-question `first_half`/`last_half` gap is still in favor of
  earlier questions; PathRelation propose at K=3 plus topical
  independence of LongMemEval questions means inter-question
  carry-over via the graph is not measurable on this bench. The
  architecture is in place; demonstrating its value needs a workload
  where consecutive queries share concepts.
- `evidence_fts` source channel does not yet show up in disabled-100
  gold-channel attribution. Distilled content already retains enough
  query keywords on LongMemEval-S that evidence-side FTS does not add
  marginal candidates in this bench. The safety net remains for
  production traces where distillation can be more aggressive.
- `graph_expansion` / `path_expansion` planes contribute 0 winning
  admissions on the three surfaces because the new usage_proof gate
  is strict and PathRelation only enters after 3 co-usage events.
  This is the design intent but means structural plane evidence
  remains future-loaded.
