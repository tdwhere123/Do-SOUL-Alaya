# v0.3.8 Closeout Report

## Status

Implementation checkpoint on branch `main` on 2026-05-16. v0.3.8 lands
seven layers of ontology mid-layer work that were previously staged as
"v0.4 future work" in protocol comments. Bench evidence on three
LongMemEval-S surfaces shows R@5 lifts on every surface, with latency
unchanged or improved.

## Scope Compliance

- MCP tool names, request schemas, and response schemas are unchanged.
  The single schema-level change is additive: `SoulOpenPointerContentSchema`
  gains optional `gist` / `excerpt` fields so EvidenceCapsule pointer
  resolution returns its raw turn material; the fields are absent on
  the MemoryEntry path.
- Protocol zod schemas (`memory-graph.ts`, `path-relation.ts`,
  `evidence-capsule.ts`) are unchanged.
- One new SQL migration (`068-evidence-capsule-fts.sql`) creates the
  evidence-side FTS5 virtual table + insert/update/delete triggers.
  No ontology table migrations.
- No runtime config schema change.

## Delivered

### Layer 1 — Object initialization

- `INITIAL_ACTIVATION_FROM_CONFIDENCE_FACTOR` bumped 0.5 → 0.6 so the
  scoring base weight has a stronger initial signal on freshly
  materialized memories. Verified rank1-rank10 score gap on
  disabled-100 lifted ~0.02 → ~0.16.
- `createContentPreview` drops the `originPlane === "global"`
  requirement. Workspace-local memories with `manifestation ===
  "full_eligible"` now serve their full content.

### Layer 2 — EvidenceCapsule + distillation

- `materialization-router.buildDistilledFact` is the new producer of
  `MemoryEntry.content`. Caller may supply
  `signal.raw_payload.distilled_fact`; otherwise a rule-based
  fallback takes up to 2 sentences capped at 280 chars across both
  Latin (.!?;) and CJK (。！？；) terminators. Raw turn lives in
  `EvidenceCapsule.gist / .excerpt`.
- `soul.open_pointer` now resolves to `EvidenceCapsule` when the
  memory lookup misses, scoped per workspace via
  `EvidenceService.findByIdScoped`. Attached agents can dereference
  `evidence_refs[i]` back to raw turn material through the same MCP
  entry point used for memory pointers.
- Migration `068-evidence-capsule-fts.sql` adds an FTS5 virtual table
  + triggers mirroring the memory-FTS pattern from migration 019.
- Recall lexical plane consults evidence FTS in addition to memory
  FTS. Evidence-side hits are resolved back to owning memories via
  `MemoryEntry.evidence_refs` (JSON `LIKE ? ESCAPE '\\'` with
  `%"<id>"%` so non-UUID ids cannot fuzz-match).

### Layer 3 — Edge producers

- The four staged `MemoryGraphEdgeType` values
  (`supersedes` / `contradicts` / `exception_to` /
  `incompatible_with`) now have writers. Caller-explicit hints via
  `raw_payload.{supersedes,exception_to,contradicts,
  incompatible_with}_refs` produce the corresponding edges at
  materialization time. The `createSourceMemoryEdges` helper was
  generalized into `createEdgesFromRawPayloadRefs` so all four hint
  types share one fire-and-forget writer.
- New `ConflictDetectionService` in `packages/core` runs at
  materialization time. Rule shape: same dimension + same scope +
  tag Jaccard ≥ 0.5 + token Jaccard < 0.35 ⇒ contradicts;
  cross-dimension or cross-scope + tag Jaccard ≥ 0.5 ⇒
  incompatible_with. Token comparator uses Unicode-aware
  tokenization across Latin and CJK.
- Optional LLM port wired via env
  (`ALAYA_CONFLICT_LLM_PROVIDER_URL` / `_API_KEY` / `_MODEL` /
  `_TIMEOUT_MS`). When configured, an openai-compatible chat
  completion request classifies each borderline neighbor as
  `contradicts` / `incompatible_with` / `none`. Hook is bounded to
  `llmMaxPairsPerNewMemory` per materialization (default 4).
- The whole service is opt-in: `ALAYA_CONFLICT_DETECTION_ENABLED=1`
  switches it on. Default off — each materialization would otherwise
  pay O(workspace_size) for `findByDimension +
  findByWorkspaceId`, which a 100-question LongMemEval seeding burst
  measured as ~150× the no-detection p95. The env-gate restored bench
  latency to baseline.

### Layer 4 — PathRelation propose

- New `PathRelationProposalService` in `packages/core`. Counts
  in-process co-usage events per memory pair; on the
  `PATH_RELATION_PROPOSE_THRESHOLD = 3` co-usage of a pair, writes a
  PathRelation with `recalls_edge_co_usage` evidence basis, initial
  strength 0.3, `bidirectional_asymmetric` direction, and
  `recall_bias 0.5`. Existing relations between the same pair are
  detected via `findByAnchors` and skipped.
- `crossLinkRecalledMemories` in `mcp-memory-tool-handler.ts` calls
  `pathRelationProposalService.onCoUsage(usedObjectIds, workspaceId)`
  after writing RECALLS edges so PathRelation propose lives alongside
  the RECALLS-edge writer at one boundary.
- `PathPlasticityService` now has a real source of paths to evolve.
  Prior to this commit PathRelation was a half-dead schema with a
  writable surface but no producer.

### Layer 5 — Recall engineering

- `task-surface-builder` default budgets widened:
  chat 10→15 entries (2000→3000 tokens),
  analyze 25→30 (4000→5000),
  build 15→20 (3000→4000),
  govern 20→25 (3500→4500).
  Targets the `budget_dropped` miss class that dominated v0.3.7
  cross-question diagnostics.
- `session_surface_cohort` seed-cohort branch gains a pre-admission
  dominance guard: when the would-be cohort (per-seed ±radius union)
  covers more than half of the tier pool, the branch is skipped so
  evidence_anchor / domain_tag_cluster / lexical compete for delivery
  instead of being drowned by trivial admit-everything cohort fanout.
- `graph_expansion` and `path_expansion` seeds now pass a
  `usage_proof` gate: a seed only projects its neighbors when it is
  either a governance-attested winner (`winnerMemoryIds`) or carries
  at least one inbound RECALLS edge. Pure lexical-match seeds no
  longer pull graph / path neighbors as structural support.

## Activation semantics

`INITIAL_ACTIVATION_FROM_CONFIDENCE_FACTOR = 0.6` is consumed in one
place: `packages/core/src/dynamics-service.ts:113`
(`activationScore = clamp01(confidence * factor)`). That callsite is
the tier-promotion path. Memories materialized through routes that do
not flow through `DynamicsService.promote*` retain whatever
`activation_score` the materializer wrote, which on the v0.3.8
materialization path is `null` until the first recall-hit-driven
promotion. Operators with workspaces that depend on promotion-tier
behavior should treat the 0.6 factor as the upper bound; the
zero-baseline below it is intentional and reflects the "memory has
not yet been used" state.

## Accept-path edge wiring — design rationale

Codex review flagged "ProposalService.accept does not write
supersedes / exception_to / PathRelation". v0.3.8 reviewed this and
concluded the indictment is based on a wrong-shape mental model:

- `ProposalSchema` (`packages/protocol/src/soul/proposal.ts`) carries
  no target / supersedes / is_exception fields.
- `acceptProposalWithDurableMemoryUpdate`
  (`apps/core-daemon/src/mcp-memory-proposal-workflow.ts:490`) calls
  `proposalRepo.acceptPendingMemoryUpdateWithEvents` which writes
  `MemoryEntryMutableFields` **in-place** on the existing
  `target_object_id`. No new memory entry is created, so no (new,
  old) memory pair forms at accept time.
- All four staged edge types (`supersedes` / `contradicts` /
  `exception_to` / `incompatible_with`) therefore must be produced
  upstream — at materialization, when the agent emits a candidate
  signal with `raw_payload.*_refs` hints, or downstream by
  `ConflictDetectionService` running rule + LLM checks against the
  newly-materialized memory's neighborhood. Both writers exist and
  are exercised by tests.

Adding an accept-time edge writer would require either a Proposal
schema extension carrying a related-memory anchor (out of v0.3.8
scope) or a synthetic memory-pair injection at accept time that
contradicts the in-place-update design choice.

## Discovered defect record (closeout truth)

### PathRelationProposalService propose path was never functional in Phase 5

The first commit that wired `PathRelationProposalService` into the
daemon (`7c269ef`) constructed the `PathRelation` with envelope-style
fields (`object_id`, `object_kind`, `schema_version`,
`lifecycle_state`) that `PathRelationSchema` (`packages/protocol/src/soul/path-relation.ts:151-177`)
strict-mode rejected. `SqlitePathRelationRepo.create` calls
`parsePathRelation` before insert. Every propose call therefore hit
the service's `try { ... } catch` block, warned, and produced no
PathRelation rows. The fix-loop round 1 (`e463b56`) replaced the
envelope-style construction with the schema-matching shape
(`path_id` + the strict field set) and added six unit tests; the fix
-loop round 2 (`0fd8623`) added a `PathRelationSchema.parse` assertion
inside one of those tests so this shape regression cannot return
silently. The defect was Blocking by severity for Phase 5 but
discovered after Phase 5 commit; recorded here so the closeout
reflects ground truth rather than the apparent green tests.

### `ruleDistillFromRaw` is a behavior change for existing workspaces

Round 1's `materialization-router.ts` change made the rule-based
distillation always cap at `DISTILLED_FACT_MAX_SENTENCES = 2`, even
when the raw turn is under the 280-char cap. Earlier behavior
returned a short raw whole. Existing `memory_entries.content` rows
written before v0.3.8 may therefore differ from what v0.3.8 would
have stored for the same input. Recall scoring reads `content`
directly, so older workspaces may exhibit different lexical match
behavior than freshly seeded ones until they are re-materialized.
Operators with persistent workspaces that depend on prior recall
shape should plan a one-time backfill / re-materialization; bench
runs do this naturally because each run seeds fresh state.

## v0.3.8 codex-review repair (round 4 + round 5)

An external adversarial review against the round-2 closeout
surfaced 3 Blocking + 7 Important findings. Round 4 landed eight
commits (`ef65f37..d548625` + `d313b6b`) closing the wiring claims;
round 5 (`0410c0d` + `0c7f80d` + `97e6d7c`) closed the verify
reviewer's residual finds. Headline source changes:

- Cohort dominance guard now runs **per-branch** (exact / seed
  branches each capped at 50% of tier pool). The round-4 union
  formulation over-skipped on single-session workspaces because
  the seed branch naturally covers all entries sharing the same
  surface_id / run_id; per-branch keeps exact (query-attested)
  admissions when seed dominates.
- Mandatory share cap in `fineAssess`: winner-attested entries
  still bypass budget unconditionally, but protected-dimension
  non-winners are now capped at `floor(max_entries * 2/3)` so the
  remaining 1/3 stays available for ranked optionals.
- Workspace-scoped pointer route: new daemon
  `GET /workspaces/:wsId/evidence/:id` using
  `EvidenceService.findByIdScoped`. Inspector proxy
  `/api/pointers/:workspaceId/:objectId` forwards to that scoped
  path instead of the unscoped `/evidence/:id`, closing the
  cross-workspace evidence read.
- LoCoMo runner R@K denominator counts only QAs with non-empty
  evidence; category-5 adversarial rows no longer deflate the
  reported R@K.
- PathRelation counter has live TTL eviction via the daemon timer
  hooked into shutdown (`intervalsToClear`), so long no-promote
  tails are bounded.
- ConflictDetectionService short-circuits when both `ruleEnabled`
  is false and no LLM port is configured, avoiding the
  O(workspace_size) fetch when neither writer can fire.
- `MEMORY_GRAPH_EDGE_RECALL_WEIGHTS` gives the v0.3.8 producers
  for `contradicts` (-0.4) and `incompatible_with` (-0.3)
  non-zero weights. They were 0 in round-2, so producer wiring
  was decorative; recall scoring now actually consumes the
  signals (suppression-only, floor-clamped at zero).

### Honest reading

The architecture work in this release — distillation,
EvidenceCapsule production, PathRelation propose, four staged
edge producers, ConflictDetectionService, Wilson CI threshold
infrastructure, LoCoMo bench wiring, Inspector Memory Browser —
**lands real wiring** for v0.3.x. The R@K signal on
LongMemEval-S is mostly the round-2 stack's; round-5 repairs
fix correctness / scope / verify-test claims without moving
the headline number.

The reasons that movement is not visible on this bench:

- LongMemEval-S questions are topically independent. PathRelation
  propose fires at K=3 co-usage of the same pair; the dataset
  does not exercise that path.
- The cohort-plane / mandatory-cap fixes change the **plane
  attribution** of delivered candidates (specifically, they stop
  cohort plane from out-competing precise candidates at rank 1)
  but the delivered set is dominated by lexical first-admit, so
  the score change is muted unless rank-1 placement is the
  binding constraint.
- `contradicts` / `incompatible_with` weights now influence
  graph_support, but those edges only exist when a workspace
  carries explicit `raw_payload.*_refs` hints or runs with
  `ALAYA_CONFLICT_DETECTION_ENABLED=1`; neither is true for the
  LongMemEval seed harness.

Where the work pays off is **non-LongMemEval workloads**:
production conversational sessions where the same memory pair
recurs (PathRelation propose actually fires), workspaces with
strong governance review (winner-attested mandatory enters the
recall path), and traces with active conflict signals
(`contradicts` weight actually flows into graph_support). The
v0.3.8 release ships these surfaces; quantitative bench gains on
those workloads is post-release follow-through.
