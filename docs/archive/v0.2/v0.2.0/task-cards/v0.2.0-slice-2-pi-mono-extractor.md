## 1. Background & Goal

**Background.** `OfficialApiGardenProvider.requestSignals`
(`packages/soul/src/garden/compute-provider.ts:140-205`) is the
daemon's only live LLM call point. It makes a hand-rolled POST to
`/v1/chat/completions` with model "gpt-4.1-mini" hardcoded and a
system prompt embedded at lines 64-70. v0.2.0 introduces pi-mono
(`@earendil-works/pi-ai`) as the LLM abstraction so the daemon stops
maintaining its own HTTP and JSON-mode plumbing.

This slice produces the extractor in isolation. The swap into
`OfficialApiGardenProvider` is the next slice; here we establish the
extractor surface, the test fixtures that pin extraction behaviour,
and the package dependency.

**Goal.** Add a `packages/soul/src/garden/pi-mono-extractor.ts` file
exporting a `SignalExtractor` interface plus a
`createPiMonoExtractor` factory backed by pi-mono. Add a
golden-dataset fixture set covering at least twenty curated turns
with expected `CandidateMemorySignal[]` outputs (these will gate the
swap in Slice 3). Add `@earendil-works/pi-ai` to `packages/soul`
dependencies.

## 2. Allowed Scope

### packages/soul/src/garden/pi-mono-extractor.ts (new)

- **Change**: export the following:
  - `interface SignalExtractor` with a single method
    `extract(input: { systemPrompt: string; userPrompt: string;
    abortSignal?: AbortSignal; timeoutMs?: number }): Promise<{
    rawJson: string }>`. The output is the raw JSON string the
    provider returned; parsing and validation stay in
    `OfficialApiGardenProvider` so the contract glue is unchanged.
  - `createPiMonoExtractor(deps: { apiKey: string; model: string;
    endpoint?: string }): SignalExtractor`. Internally constructs a
    pi-mono `Model` via `getModel(provider, modelId)` (provider
    defaults to OpenAI-compatible; `endpoint` override is passed to
    pi-mono's base URL configuration). Calls `model.complete(...)`
    with the system + user prompt, requests JSON response_format,
    and returns the text payload.
- **Errors**: surface a `SignalExtractorError` union covering
  `timeout`, `transport_failure`, and `invalid_json` (mirroring the
  failure modes the current `requestSignals` reports). Re-use
  Garden's existing error mapping conventions; do not invent new
  shapes.
- **No system prompt in this file**: the prompt lives at
  `compute-provider.ts:64-70` and stays there until a future release
  decides to revise it. The extractor receives `systemPrompt` as an
  input argument.

### packages/soul/src/__tests__/fixtures/garden-extraction-golden/ (new)

- **Change**: add at least twenty fixtures. Each fixture is a folder
  named `<NN-short-slug>/` containing:
  - `turn.txt` — a curated turn excerpt (English or Chinese as
    needed; localized content is allowed per CLAUDE.md `File Rules`)
  - `expected.json` — the expected
    `readonly CandidateMemorySignal[]` shape after parsing, with
    `signal_kind`, `object_kind`, `confidence` set; non-critical
    fields like `raw_payload` omitted.
- The fixture choice MUST include: clear naming-preference signals,
  multilingual content, ambiguous low-confidence cases, and at least
  one "no signals" case (`expected.json` is an empty array).
- Add `packages/soul/src/__tests__/garden-extraction-golden.test.ts`
  that loads each fixture, invokes the extractor against a **stub
  pi-mono model**, asserts the wired-up extractor returns the raw
  JSON that the stub provided. This test pins the extractor's
  pass-through behaviour without making real network calls.

### packages/soul/package.json

- **Change**: add `@earendil-works/pi-ai` to `dependencies` at the
  version pin agreed with the upstream README (record the exact
  version selected in this card's completion report). Update the
  lockfile with `rtk pnpm install` and commit the diff atomically.

## 3. Deferred

- The system-prompt rewrite is an intentional v0.2.0 non-goal from
  Decision B. Do not open backlog until a concrete prompt-quality
  target and acceptance evidence are approved.
- Streaming `extract` (event-driven, not buffered) is deferred; the
  Garden compile path is single-shot extraction, so the
  `complete()` variant is sufficient. Do not queue streaming work
  unless a provider or latency target requires partial extraction.
