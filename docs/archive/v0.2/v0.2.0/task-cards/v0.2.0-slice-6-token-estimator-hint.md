## 1. Background & Goal

**Background.** Two duplicate token-cost estimators exist:
`recall-service-helpers.ts:313-315` and
`context-lens-assembler.ts:688` both compute
`Math.ceil(content.length / 4)`. The estimate is provider-agnostic
and shape-blind. Recall budget shaping (Slice 5) just made budget
penalty more sensitive to actual utilization; without a better
token estimate the budget shape is still naive.

The daemon does not own a real tokenizer (and Decision D-2 of the
plan explicitly forbids shipping a native tokenizer — zero-config
posture). Instead, the host (Codex / Claude Code) is best placed to
know the model in use. v0.2.0 exposes an MCP-level *hint* that
clients can populate when they know the tokenizer family; absent the
hint, the existing `len / 4` fallback stays byte-identical.

**Goal.** Add an optional `host_context` field on the `soul.recall`
MCP input carrying `tokenizer_hint` and `host_context_window`.
Introduce a `TokenEstimator` interface and thread the per-call hint
through `RecallService.recall(...)`'s params (NOT through a
stateful `RecallServiceDependencies` field — `RecallService` is a
boot-time singleton, so a hint-bound estimator on the constructor
would be ignored or bleed across calls; at most the constructor may
hold a *stateless* `TokenEstimatorFactory`). Unify the two
duplicate `len / 4` implementations behind the per-call estimator.

## 2. Allowed Scope

### MCP input schema for `soul.recall`

- **Locate** the existing input schema (search
  `rg -n 'soul.recall' packages/protocol/src/soul`).
- **Change**: add optional
  ```ts
  host_context: z.object({
    tokenizer_hint: z.enum([
      "cl100k", "o200k", "approx_chars_per_token"
    ]).optional(),
    host_context_window: z.number().int().positive().optional()
  }).optional()
  ```
- The field is additive on a public MCP surface — §25 SemVer minor
  bump. The field is fully optional including its nested fields,
  so any existing caller continues to type-check without change.

### packages/core/src/recall-service-types.ts (new file)

- **Change**: export
  ```ts
  export interface TokenEstimator {
    estimate(text: string): number;
  }
  export function makeTokenEstimator(opts: {
    hint?: "cl100k" | "o200k" | "approx_chars_per_token";
  }): TokenEstimator;
  ```
- `makeTokenEstimator`:
  - `"cl100k"`: heuristic ratio 3.6 chars per token
    (rough OpenAI cl100k-base average for English; document the
    citation in a code comment)
  - `"o200k"`: heuristic ratio 3.2 chars per token
  - `"approx_chars_per_token"`: same as the fallback (4)
  - default (no hint): `Math.ceil(text.length / 4)` — byte-identical
    to v0.1 behaviour

### packages/core/src/recall-service-helpers.ts (line 313)

- **Change**: delete the inline
  `estimateTokens(content) = Math.ceil(content.length / 4)` and
  replace the calling site with a `tokenEstimator.estimate(content)`
  call passed in via the service dependencies. Update the helper's
  signature to accept the estimator (or inline the call from the
  callsite).

### packages/core/src/context-lens-assembler.ts (line 688)

- **Change**: same migration — replace the inline estimator. The
  assembler receives the per-call `TokenEstimator` from the recall
  call that drives it (passed down from `RecallService.recall`), not
  from a boot-time constructor dependency.

### RecallService per-call estimator (NOT a boot-time dependency)

`RecallService` is constructed once at daemon boot
(`apps/core-daemon/src/index.ts`, around lines 479-505), so a
`tokenizer_hint` that varies per `soul.recall` call cannot ride on a
constructor dependency — it would be ignored or bleed across calls.
The estimator must be **per-call**:

- **Locate** `RecallService.recall(...)` params
  (`packages/core/src/recall-service.ts`, around lines 94-101) and
  the MCP handler call site (`apps/core-daemon/src/mcp-memory-tool-handler.ts`,
  around lines 347-354 where `deps.recallService.recall(...)` is
  invoked).
- **Change**: thread `host_context` (or just the resolved
  `tokenizer_hint`) through `RecallService.recall`'s params. Inside
  `recall`, build the `TokenEstimator` for this call via
  `makeTokenEstimator({ hint })` and pass it down to the helpers and
  the context-lens assembler.
- **Alternative (acceptable)**: inject a `TokenEstimatorFactory`
  (constructor dependency) and call `factory.forHint(hint)` per
  recall. Either way the *instance used* is per-call; what the
  constructor holds is at most a stateless factory.
- The constructor dependency set does NOT gain a stateful
  `tokenEstimator` instance.

### Tests

- **Add** a small unit test that:
  - asserts the no-hint path returns `Math.ceil(text.length / 4)`
    byte-identical to a v0.1 fixture
  - asserts each hint returns a *smaller-or-equal* count than the
    no-hint fallback for a 100-character English string (mild
    sanity check)
- **Update** any test that mocked the helper-level estimator
  directly (most production tests stub at the recall-service
  boundary so disruption should be small).

## 3. Deferred

- Real tokenizer integration (tiktoken-node, sentencepiece) is
  not queued; the zero-config posture is the v0.2 product invariant.
- `host_context_window` consumer logic is no longer a live deferral:
  the field was removed before v0.2.0 release acceptance. Today only
  `tokenizer_hint` flows into the estimator.
