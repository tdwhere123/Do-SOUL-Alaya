## 1. Background & Goal

**Background.** `OfficialApiGardenProvider.requestSignals` is the
daemon's only LLM call point. After Slice 2 the
`createPiMonoExtractor` factory exists and is golden-dataset-pinned.
This slice replaces the hand-rolled HTTP path inside
`OfficialApiGardenProvider` with the extractor; the system prompt
(lines 64-70) stays unchanged so any extraction-quality drift
between the legacy `fetch` path and the pi-mono path is visible
through the golden-dataset diff.

**Goal.** Refactor `OfficialApiGardenProvider` to accept an injected
`SignalExtractor`, switch the production default to the pi-mono
extractor, and prove signal-extraction parity via a swap-before /
swap-after golden-dataset comparison.

## 2. Allowed Scope

### packages/soul/src/garden/compute-provider.ts

- **Change**: `OfficialApiGardenProviderDependencies` gains an
  optional `extractor?: SignalExtractor`. The constructor's default
  builds `createPiMonoExtractor({ apiKey, model, endpoint })` from
  the existing constructor inputs.
- **Change**: `requestSignals` body now calls
  `extractor.extract({ systemPrompt, userPrompt, abortSignal,
  timeoutMs })`. Replace the `fetch(...)` call and its surrounding
  abort/timeout plumbing with the extractor invocation. Keep
  `parseOfficialApiSignals`, `clampConfidence`, `buildTurnExcerpt`,
  and the `CandidateMemorySignalSchema.parse` boundary unchanged.
- **Change**: drop now-unused dependency fields from
  `OfficialApiGardenProviderDependencies` (`fetchImpl`,
  `createAbortController`, `setTimeoutImpl`,
  `clearTimeoutImpl`) **only after** the call sites that rely on
  them are migrated to inject a fake extractor. If any test fixture
  still passes one of these dependencies, migrate the fixture in
  the same commit.
- **Keep**: `OFFICIAL_API_SYSTEM_PROMPT` at lines 64-70. Do not
  rewrite it in this card.
- **Keep**: the priority rank in
  `compute-routing-service.ts:24-29` (`OFFICIAL_API=0` etc.) — this
  card does not change routing.

### packages/soul/src/__tests__/compute-provider.test.ts (and any hoist-mocking sites)

- **Change**: update test fixtures that mocked `OfficialApiGardenProvider`'s
  HTTP plumbing (e.g. `apps/core-daemon/src/__tests__/tool-runtime-wiring-fixture.ts:1104`
  if it still hoist-mocks the class). Keep the class name and the
  `GardenComputeProvider` interface unchanged so the mock surface
  remains stable.

### packages/soul/src/__tests__/garden-extraction-parser-parity.test.ts (new)

- **Change**: a new test loads each golden fixture from
  `garden-extraction-golden/`, drives `OfficialApiGardenProvider` with
  a `SignalExtractor` stub that returns the recorded raw JSON for
  that fixture, and asserts the parsed `CandidateMemorySignal[]`
  matches the expected output **byte for byte** on `signal_kind` and
  `object_kind` (`confidence` ±0.05). This test proves **parser
  parity only** — it does NOT exercise `createPiMonoExtractor` and
  does NOT prove the production provider sends the same request or
  gets the same JSON. Name it accordingly so nobody mistakes it for
  end-to-end parity.

### packages/soul/src/__tests__/pi-mono-extractor-contract.test.ts (new)

- **Change**: a transport-level contract test for
  `createPiMonoExtractor`. Inject a fake pi-mono `Model` whose
  `complete(...)` records the request it received. Assert:
  - the extractor passes through the exact `systemPrompt`
    (`OFFICIAL_API_SYSTEM_PROMPT` from `compute-provider.ts:64-70`,
    unchanged) and the exact `userPrompt` it was given
  - the extractor requests JSON-mode / structured output
    (whatever pi-mono's `response_format` knob is) so the provider
    still gets a parseable JSON payload
  - `model`, `endpoint` (base URL), `abortSignal`, and `timeoutMs`
    reach pi-mono unchanged
  - on a non-JSON response the extractor surfaces
    `SignalExtractorError` with `kind === "invalid_json"`, on
    timeout `kind === "timeout"`, on transport failure
    `kind === "transport_failure"` — matching the failure modes the
    legacy `requestSignals` reported
- This is the gate that actually catches "pi-mono sends a different
  request than the old fetch did". The parser-parity test above only
  catches "the parser still accepts the fixture shape".

### Legacy fetch cassette (optional but recommended)

- **Change**: if a recording of the pre-swap `fetch` request to
  `/v1/chat/completions` can be captured (headers + body), commit it
  under `packages/soul/src/__tests__/fixtures/legacy-garden-fetch-cassette.json`
  and assert in `pi-mono-extractor-contract.test.ts` that the
  pi-mono path produces an equivalent request (same model, same
  messages array, same JSON-mode directive). If the cassette cannot
  be captured cleanly, document why in the completion report and
  rely on the contract assertions above.
