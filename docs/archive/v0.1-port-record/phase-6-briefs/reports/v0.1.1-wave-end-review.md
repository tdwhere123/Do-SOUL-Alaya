## Important Findings (must fix this wave)

### M1 — `garden.complete_task` lacks scope-strip on candidate_signals (§29)

**Source:** Spec F1 (`apps/core-daemon/src/mcp-memory-tool-handler.ts:514-520`).

`GardenTaskCompleteRequestSchema.result_envelope.candidate_signals` accepts
the full `CandidateMemorySignalSchema`, which carries `workspace_id`,
`run_id`, `surface_id`, and `source` fields. The handler validates only
`workspace_id`. A host can therefore submit completion with
`source = "user_seed"` or with a fabricated `run_id` that does not match
the claimed task — the same payload-spoof the v0.1.0 commit `e6378dd`
hardened on `soul.emit_candidate_signal`.

**Fix:** strip server-bound scope at schema time the same way I5 did for
emit_candidate_signal. Either fork a content-only schema for
`result_envelope.candidate_signals` or have the handler rebuild the
signal from `(claimed task row) + (trusted MCP context) + (host-provided
content fields only)` before dispatching to `signalService.receiveSignal`.

### M2 — `garden.complete_task` partial-success window (§10) [Spec F2 / Reviewer I1]

**Source:** Spec F2 + Reviewer I1
(`apps/core-daemon/src/mcp-memory-tool-handler.ts:520-555`).

`completeWithEvents` atomically commits the `SOUL_GARDEN_TASK_COMPLETED`
event row whose payload includes `objects_affected = signal_ids[]` and
`candidate_signals_count = N`. Only **after** the commit does the handler
loop `signalService.receiveSignal` for each candidate. If any
`signalService.receiveSignal` throws (validation reject, unique-id
collision, transient error), the audit trail claims N signals were
emitted but `soul.signal.emitted` rows are absent. Reverse §10
"audit-precedes-broadcast" — the audit row precedes a broadcast that
never happens.

**Fix:** move the `receiveSignal` loop INTO the `completeWithEvents`
mutate callback so the `SOUL_GARDEN_TASK_COMPLETED` event AND every
resulting `soul.signal.emitted` event share one
`appendManyWithMutation` transaction. If receiveSignal throws,
the entire batch rolls back including the task-completed event, and the
host gets a clean error.

### M3 — H3 dedupe relies on fragile string match (Reviewer I2)

**Source:** Reviewer I2 (`apps/core-daemon/src/mcp-memory-tool-handler.ts:`
`isDuplicatePostTurnExtractTask` helper).

`isDuplicatePostTurnExtractTask` walks the cause chain looking for
`.includes("garden_tasks.id")` in error messages. better-sqlite3's
constraint error text format is library-version-coupled. The codebase
already standardised structured CAS detection in commit `aacb4f2`
(I3 fix added `StorageError("DUPLICATE_KEY", ...)` for workspace-create
collisions).

**Fix:** raise `StorageError("DUPLICATE_KEY", ...)` from
`SqliteGardenTaskRepo.enqueue` when the SQLITE_CONSTRAINT fires on
`garden_tasks.id`; check `error.code === "DUPLICATE_KEY"` in
`isDuplicatePostTurnExtractTask`. Mirrors the workspace-repo CAS
predicate.

### M4 — R2 vs H3 disagree on "is used" (Reviewer I3)

**Source:** Reviewer I3
(`apps/core-daemon/src/mcp-memory-tool-handler.ts` — R2 path vs H3
enqueue path within the same `report_context_usage` handler).

Inside one `report_context_usage` invocation:

- R2 (tier promotion) reads `delivered.usage_state === "used"`.
- H3 (POST_TURN_EXTRACT enqueue) reads `delivered_objects` array
  contents.

Same untrusted request, two divergent interpretations of "used". Edge
cases where the two diverge produce ghost tier promotions OR ghost
extract tasks.

**Fix:** unify the predicate at the top of the handler — compute one
canonical `usedDeliveredObjects` collection from the parsed
`SoulReportContextUsageRequest` and have both R2 and H3 consume it.

### M5 — R1 cascade calls `collectSupplementaryData` 3× in cold start (Reviewer I4)

**Source:** Reviewer I4 (`packages/core/src/recall-service.ts` — cascade
loop).

When HOT yields zero candidates, R1 cascades:

1. Coarse-filter HOT → fineAssess → empty.
2. Cascade WARM → coarse-filter → fineAssess → still empty.
3. Cascade COLD → coarse-filter → fineAssess.

Each `fineAssess` calls `collectSupplementaryData`, which fans out a
batch of N+1 lookups across plasticity, graph, and embedding ports.
Cold-start is exactly the scenario R1 was built for, so 3× the supplement
fan-out lands on the worst path.

**Fix:** hoist `collectSupplementaryData` to gather the union of HOT +
WARM + COLD candidate IDs once per recall, after the cascade decision is
made. Fine-assess then runs against the prefetched supplementary data.

### M6 — H1 dispatch claim and audit event not in same tx (Reviewer I5)

**Source:** Reviewer I5
(`packages/soul/src/garden/scheduler.ts` — `dispatch` flow;
`packages/storage/src/repos/garden-task-repo.ts` — `claimAtomic`).

`scheduler.dispatch` flow:

1. `repo.claimAtomic(taskId, "in-process", now)` → SQL CAS
   commits `status=claimed`.
2. Worker callback runs.
3. `repo.completeWithEvents(taskId, ..., events)` commits
   `SOUL_GARDEN_TASK_COMPLETED`.

The `SOUL_GARDEN_TASK_DISPATCHED` event is only published **after** the
worker completes. If the daemon crashes between step 1 and step 3, the
DB row reads `status=claimed, claimed_by="in-process"` but no
`SOUL_GARDEN_TASK_DISPATCHED` event was written. Recovery relies on
`gcAbandonedClaims` — correct, but §10 prefers the dispatched event to
be on disk before any side-effecting work runs.

**Fix:** merge the dispatched event into the same SQLite transaction
as `claimAtomic`. Repo method becomes
`claimAtomicWithEvents(taskId, claimedBy, claimedAt, dispatchedEvents)`;
`UPDATE ... + INSERT INTO event_log` are one tx.

### M7 — H1 100-task / 8-claimer race test does not simulate concurrency (Reviewer I6)

**Source:** Reviewer I6
(`apps/core-daemon/src/__tests__/garden-task-repo-cas.test.ts`).

The race test uses `Promise.all` of 8 claimer loops against a shared
better-sqlite3 connection. better-sqlite3 is single-threaded
synchronous; Promise.all serialises the 800 claim attempts on one
thread, so the test passes because `UPDATE ... WHERE status='pending'`
is intrinsically atomic against itself, not because Promise.all
exercises concurrency. Diagnostic value of the test is lower than the
file size suggests.

**Fix:** either (a) augment with a worker-threads multi-connection test
that does fork a real race; (b) replace with a SQL-level invariant test
(`SELECT COUNT(*) WHERE status='claimed' GROUP BY id` → all counts = 1)
documenting that better-sqlite3's single-threaded model is sufficient
proof; OR (c) keep the test but document the limitation in a comment so
future readers understand what it does and does not prove.

## Nice-to-have Findings

### N1 — Dual `SqliteGardenTaskRepo` instances (Spec F3)

`apps/core-daemon/src/index.ts:697-700` and
`apps/core-daemon/src/garden-runtime.ts:200` each `new` a
`SqliteGardenTaskRepo` against the same `SqliteConnection`. CAS still
holds because the lock is row-level. Future readers will find the dual
instantiation confusing — either wire one and reuse, or document why
two are intentional.

### N2 — `garden.complete_task` does not assert claimed_by matches context (Spec F4)

When two host agents (e.g., `codex` + `claude-code`) attach to the same
workspace and one claims a task, the other can complete it as long as
the task is in `status=claimed`. Low risk in practice (single-host is
the dominant scenario), but worth a `claimed_by === ctx.agentTarget`
check so a complete from a different agent target returns
`already_completed_by_other`.
