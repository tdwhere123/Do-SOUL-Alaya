## 1. Background & Goal

**Background.** Today `mcp-callable` is proven only by an SDK-driven
deterministic harness (`apps/core-daemon/src/__tests__/agent-use-protocol.test.ts`):
the daemon serves `soul.*` calls when something invokes them. It does
*not* prove that a real Codex or Claude Code session, with the
attach-written `operator_instructions`, autonomously selects
`soul.recall` / `soul.report_context_usage` during a normal user
conversation. The telemetry shape (`soul.recall.delivered`,
`soul.context_usage.reported`) and aggregation (`alaya status
--recall-stats`) shipped in v0.1.1; what is missing is a recorded
live-host transcript and a regression that keeps the daemon contract
observable across host model upgrades.

**Goal.** Record one real Codex (or Claude Code) session in which the
host autonomously calls `soul.recall` and `soul.report_context_usage`,
land the transcript + pre-seeded state + host-version metadata as a
fixture, and add an offline replay regression that re-derives the
same EventLog rows from the fixture without a live host.

## 2. Allowed Scope

### docs/archive/v0.3-historical/v0.3.0/host-autonomy-fixtures/<host>-<version>/ (new)

- **Change**: a fixture directory per recorded `(host, version)`
  pair containing: the MCP stdin transcript of the session; the
  pre-seeded daemon state (or a script that produces it — fresh
  install in an isolated config dir, fresh attach, no pre-seeded
  memory beyond what the recall-worthy scenario needs); fixture
  metadata recording the host name and exact CLI version, the
  attach-written `operator_instructions` snapshot, and the prompt
  used. Capture conditions follow `#BL-038`'s acceptance list (fresh
  install → fresh attach → real MCP stdio session → recall-worthy
  prompt → at minimum one `soul.recall.delivered` and one
  `soul.context_usage.reported` with `usage_status == "used"`, both
  originating from the host, not a test harness).

### apps/core-daemon/src/__tests__/host-autonomy-replay.test.ts (new)

- **Change**: boot a daemon with the fixture's pre-seeded state,
  replay the recorded MCP stdin transcript, and assert the daemon
  produces the same EventLog rows modulo non-deterministic
  identifiers (timestamps, `delivery_id`, etc.) — at minimum one
  `soul.recall.delivered` and one `soul.context_usage.reported` with
  `usage_state == "used"`. The test runs offline; no live host is
  required. The test header documents the ageing-out rule: when the
  upstream host model changes its autonomous tool-selection
  behaviour, the *recording* ages out and a fresh one is captured,
  but this offline replay still pins the daemon contract.

### docs/archive/v0.3-historical/v0.3.0/reports/v0.3.0-slice-4.md (new)

- **Change**: completion report (host + version recorded, capture
  conditions, the EventLog evidence, any deviations; if the fallback
  path was taken — see §3 — record which and why).

## 5. Verification

```bash
rtk pnpm build
rtk pnpm exec vitest run --project @do-soul/alaya-core-daemon -- host-autonomy-replay
rtk pnpm exec tsc -b
ls docs/archive/v0.3-historical/v0.3.0/host-autonomy-fixtures/
```

Recording procedure (manual, produces the fixture): fresh install in
an isolated config directory → fresh attach (no pre-seeded memory) →
start a real MCP stdio session against the target Codex / Claude Code
CLI version → run a recall-worthy prompt (e.g. "what was the user's
stated naming preference for retry helpers?") → capture EventLog rows
from the daemon (`soul.recall.delivered` + `soul.context_usage.reported`,
`usage_status == "used"`, host-originated) → save transcript +
metadata under the fixture directory.
