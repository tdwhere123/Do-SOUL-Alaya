# v0.3.0 Slice 4 Report - Host Autonomy Witness (#BL-038)

## Status

RESOLVED via live-usage witness. The original strict gate (fresh-install +
recorded raw MCP stdio transcript + offline replay) was dropped as
over-engineering after the operator pointed out that the authentic test is the
operator's own normal Codex / Claude Code conversations, which the attached
daemon already records as the required chain. v0.3.0 closes `#BL-038` with this
witness rather than carrying it forward.

(Supersedes the earlier BLOCKED report. The "no `usage_state == "used"` row"
finding in that version was stale within a day — the live EventLog accumulated
six such chains from real Claude Code sessions; see Evidence below.)

## What changed vs. the original slice-4 plan

- The "synthetic transcript / offline replay" requirement is removed. The
  daemon does not record raw MCP stdio frames in production, and a hand-built
  transcript is exactly the fabricated proof `#BL-038` rejects. Instead the
  witness is a snapshot of real attached-host EventLog rows.
- `docs/archive/v0.3-historical/v0.3.0/host-autonomy-fixtures/README.md` rewritten to define the
  live-usage-witness fixture shape.
- `scripts/export-host-autonomy-witness.mjs` added — reads a live Alaya
  EventLog (`$ALAYA_CONFIG_DIR/alaya.db` or `~/.config/alaya/alaya.db`), picks
  every complete `soul.recall.delivered` (`pointer_count >= 1`) →
  `soul.context_usage.reported` (`usage_state == "used"`) chain linked by
  `delivery_id`, and writes `event-log.jsonl` + `metadata.json` under
  `host-autonomy-fixtures/<host>-live/`. Re-runnable.
- `apps/core-daemon/src/__tests__/host-autonomy-witness.test.ts` added — reads
  the committed fixture and asserts the chain shape offline.
- Trust-surface wiring fixed alongside (was: `alaya attach` wrote the MCP
  server entry without a host identity, so recall/usage telemetry was bucketed
  under the generic `mcp` `agent_target` and the `claude-code` / `codex` trust
  counters in `alaya status` stayed at 0): `alaya attach claude` / `alaya
  attach codex` now stamp `ALAYA_AGENT_TARGET` into the launched `alaya mcp
  stdio` env, which `resolveMcpAgentTarget` already accepts.

## Evidence captured

`docs/archive/v0.3-historical/v0.3.0/host-autonomy-fixtures/claude-code-live/` — 6 chains, 12
events, exported from the operator's live `~/.config/alaya/alaya.db`. All from
real Claude Code MCP stdio sessions (`mcp-session-*` runs); `agent_target=mcp`
because the rows pre-date the v0.3.0 attach env stamp. The
host-autonomy-witness test is green.

`alaya status` showed `claude-code: ... delivered=0 used=0` before the fix
because the rows were attributed to `agent_target=mcp`, not `claude-code`. The
attach env-stamp fix corrects future rows; the operator should re-run `alaya
attach claude` / `alaya attach codex` on the v0.3.0 daemon and re-run the
export script to refresh the fixture with host-labelled rows.
