# v0.3.0 Slice 3 Report - `install --keychain`

## Status

CODE COMPLETE, but RELEASE BLOCKED.

The interactive install path is implemented and tested in the controller
worktree. v0.3.0 cannot close because the required Linux, macOS, and Windows
platform transcripts have not been captured.

## Work Completed

- Added platform write and availability adapters:
  - macOS: `security add-generic-password -U`
  - Linux: `secret-tool store`
  - Windows: PowerShell `Windows.Security.Credentials.PasswordVault`
- Kept platform keychain write secrets out of process argv. Linux and Windows
  write secrets through stdin; macOS uses `security -i` and sends the
  `add-generic-password` command through stdin instead of passing the secret as
  a spawned argument.
- Added `install --keychain` as an interactive-only install path.
- Refused `install --keychain --non-interactive` with a specific usage error;
  no env or JSON secret shortcut was added.
- Masked the TTY secret prompt path; the test harness exercises the TTY branch
  and asserts the secret is absent from stderr.
- Wrote the dedicated Garden keychain ref
  `ALAYA_OFFICIAL_GARDEN_SECRET_REF=keychain:alaya:openai`.
- Preserved the existing embedding `ALAYA_OPENAI_SECRET_REF` line during
  migration.
- Migrated an existing persisted `runtime:garden-compute` config row to the
  keychain ref when that row exists, so DB-backed runtime config does not keep
  routing through the old `env:` / `file:` ref after install success.
- Kept the existing failed-install audit guard in front of keychain writes;
  `--force` remains required to override unrecovered partial install state.
- Verified the written keychain entry before patching `.env`.
- Kept secret values out of stdout, stderr, `.env`, and audit JSON.
