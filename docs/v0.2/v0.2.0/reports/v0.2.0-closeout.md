## Review/Fix Loop

Round 1 found Blocking / Important issues. Local fixes landed for:

- v0.2.0 package metadata drift (`0.1.3` -> `0.2.0`).
- Runtime Garden compute PATCH now refreshes routing candidates without
  daemon restart.
- `source_delivery_ids` are validated against recorded deliveries in the
  current trusted context before agent-originated signal/proposal writes.
- The SemVer snapshot now hashes schema signatures, not only object keys.
- Runtime/release docs no longer claim credentialed pi-mono live smoke
  proof in this environment.

Round 2 found Blocking / Important issues. Local fixes landed for:

- Direct `createMcpMemoryProposalWorkflow().proposeMemoryUpdate()`
  calls now fail closed when `source_delivery_ids` are supplied without
  an injected trust-state validator; public MCP and direct workflow
  paths both have negative forged-anchor coverage.
- Garden compute provider default resolution is now dynamic: no-modelRef
  runtime paths resolve the current default routing candidate after a
  hot config PATCH instead of retaining the boot-time provider object.
- v0.2 README layout/status language now says candidate surface and
  keeps PASS blocked on credentialed live pi-mono smoke evidence.

Round 3 read-only re-review cleared local Blocking / Important issues
for proposal anchor validation, Garden default provider refresh, and
v0.2.0 candidate status language.

Round 4 (pre-merge multi-lens re-review on the integrated branch) found
no Blocking issues. Local fixes landed for the Important findings:

- `parseOfficialApiSignals` and the pi-mono extractor now bound a
  hostile/garbage official-provider response: at most 64 signals, and
  `object_kind` / `matched_text` / `reason` are length-clamped (200 /
  4000 / 400 chars); the extractor rejects a response body over
  256 000 chars as `invalid_json`. Garden runs fire-and-forget on every
  turn, so an operator-misconfigured or hijacked `provider_url` can no
  longer amplify into unbounded EventLog writes.
- The runtime Garden-compute config-change audit
  (`SOUL_HEALTH_JOURNAL_RECORDED` → `change_summary`) now records the
  new non-secret `provider_url` and `model_id` values, not only the
  changed field names. A hot `PATCH /config/runtime/garden-compute`
  that repoints the provider URL is now actionable in the event log.
  `secret_ref` stays reduced to `secret_ref_kind`.
- The `garden-extraction-golden` test gained a header comment stating it
  proves transport shape, not extraction quality; extraction quality is
  gated by the live pi-mono smoke recorded for Slice 3.

Round 4 follow-up (the Nice-to-haves Round 4 deferred — all resolved on
`main` in the post-merge v0.2.x follow-up commit; no release blockers,
done immediately rather than carried):

- `source_delivery_ids` arrays now cap at 32 anchors
  (`SourceDeliveryIdsSchema` in `candidate-memory-signal.ts` /
  `events/signal.ts` / `events/memory-governance.ts`, plus the inline
  `SoulProposeMemoryUpdateRequestSchema.source_delivery_ids` array), so
  one MCP call can no longer fan out into an unbounded sequence of
  `findDeliveryById` lookups. Additive constraint within the same
  unreleased v0.2.0 surface.
- `ConversationService.resolveGardenComputeProvider` now falls back to
  `resolveGardenComputeProvider.resolve(null)` (the current default
  routing candidate) when the requested model ref does not match, so a
  hot config PATCH is reflected on the next turn; the boot-time
  `gardenComputeProvider` captured in `apps/core-daemon/src/index.ts` is
  now only the last-resort backstop when no resolver is wired (never in
  production).
- `GardenComputeProviderResolver.provider_kind` is now a getter backed
  by `lastResolvedKind`: it starts at the resolver's nominal
  `OFFICIAL_API` and `getProvider()` corrects it to the kind it actually
  resolved, so a routing tie-breaker on `provider.provider_kind` is
  accurate when the resolver is serving the local-heuristics fallback.
- Removed the orphaned post-slice-1 stance/routing code with no
  production caller: `apps/core-daemon/src/compute-routing-resolver.ts`,
  `packages/core/src/stance-resolution-service.ts` (and its
  `@do-soul/alaya-core` re-export), and `createStancePolicyProvider`
  from `apps/core-daemon/src/daemon-runtime-helpers.ts`, plus their
  tests and the wiring-fixture mock. `code-map.md` updated.
- Removed the unused `host_context_window` field from
  `SoulRecallHostContextSchema` before it ossified; `host_context` still
  carries `tokenizer_hint`, which recall token estimation consumes.

Also: `alaya inspect` now auto-selects the workspace whose registered
`repo_path` resolves to the current directory when several workspaces
are active (it still lists candidates and asks for `--workspace <id>`
when zero or more than one match), so running `alaya inspect` from a
project root no longer requires the explicit flag.

`.do-it/artifacts/alaya-0.1.1-live-system-test/alaya-api.env`). Driver:

`createPiMonoExtractor` path internally — so it exercises the pi-mono

`CandidateMemorySignalSchema.parse` boundary against a real
OpenAI-compatible endpoint. Result: 3 well-formed `CandidateMemorySignal`s
with consistent `signal_kind` / `object_kind`, ~3.9 s round trip, no error.
Full transcript and rerun command in
`docs/v0.2/v0.2.0/task-cards/reports/v0.2.0-slice-3.md`. Not exercised:
the full daemon `POST_TURN_EXTRACT` task lifecycle and the EventLog row,
