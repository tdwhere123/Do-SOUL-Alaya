## 1. Background & Goal

**Background.** Boot-time secret resolution for
`OfficialApiGardenProvider` is already wired through
`resolveGardenOpenAiCredential`
(`apps/core-daemon/src/garden-credential.ts`). When an operator
PATCHes `RuntimeGardenComputeConfig.secret_ref` (or `model_id` /
`provider_url`) via the Inspector, the new value lands in
`configRepo` but the provider instance keeps its old construction
arguments. This silent staleness was acceptable when secret rotation
was rare; with pi-mono now the live LLM path, runtime config patch
is the explicit operator surface and the daemon MUST honour it
without a restart.

**Goal.** Add a `GardenComputeProviderResolver` that lazily reads
the current `RuntimeGardenComputeConfig` from `configRepo` on each
compile invocation, resolves `secret_ref` via the existing
`resolveSecretRef` helper, and caches the
`OfficialApiGardenProvider` instance keyed by `(secret_ref,
model_id, provider_url)`. The Inspector PATCH path invalidates the
cache.

## 2. Allowed Scope

### apps/core-daemon/src/services/garden-compute-provider-resolver.ts (new)

- **Change**: export
  `class GardenComputeProviderResolver` with:
  - constructor `(deps: { configRepo, secretReader, makeProvider:
    (resolvedConfig) => GardenComputeProvider })`. `secretReader`
    encapsulates `resolveSecretRef` to keep the resolver
    test-friendly; `makeProvider` is the factory hook for tests.
  - method `async getProvider(): Promise<GardenComputeProvider>`
    that:
    1. reads the current `RuntimeGardenComputeConfig`,
    2. computes cache key `<secret_ref>|<model_id>|<provider_url>`,
    3. returns the cached provider if key matches; otherwise
       resolves the secret, builds a new provider via
       `makeProvider`, replaces the cache entry, and returns it.
  - method `invalidate(): void` that drops the cached instance and
    its key. Called by the runtime-config PATCH path (see below).
- **Errors**: when the resolved secret is malformed or the platform
  refuses to read it, surface a typed error consistent with
  Garden's existing failure modes; the caller logs and falls back
  to `LocalHeuristics` per the existing routing service. The
  resolver itself does not swallow errors.

### apps/core-daemon/src/index.ts

- **Change**: at the existing Garden provider construction site
  (line 594), replace the eager `new OfficialApiGardenProvider(...)`
  with `const gardenComputeProviderResolver = new
  GardenComputeProviderResolver({...})`. Pass
  `gardenComputeProviderResolver.getProvider.bind(...)` (or the
  resolver itself) into the routing service / consumers that
  previously received the eager instance. Verify no other consumer
  retains a stale reference.
- **Change**: when the runtime-config PATCH handler accepts a
  `garden.compute.*` mutation, call `resolver.invalidate()` after
  the configRepo write commits. This may already live in the
  Inspector PATCH path; locate the call site and add the
  invalidation. Do not add a generic post-write hook to configRepo
  — that crosses ownership and v0.1 left configRepo as a passive
  store.

### apps/core-daemon/src/__tests__/garden-compute-provider-resolver.test.ts (new)

- **Change**: tests covering:
  - first call constructs a provider; second call with same config
    returns the cached instance.
  - changing `secret_ref` (via the configRepo) on a subsequent call
    rebuilds the provider.
  - `invalidate()` drops the cache; the next call rebuilds.
  - error path: malformed `secret_ref` propagates a typed error;
    the resolver does NOT cache the failure (so a corrected PATCH
    becomes observable on the next call).

### apps/core-daemon/src/__tests__/garden-runtime-config-patch.test.ts (extend if exists; else new)

- **Change**: end-to-end check that a PATCH on
  `RuntimeGardenComputeConfig.secret_ref` triggers
  `resolver.invalidate()` and the next compile invocation reads the
  new credential. Stub the secret reader so the test is
  deterministic.

## 3. Deferred

- Multi-tenant cache (per-workspace overrides) — Alaya v0.2 ships a
  single-process daemon with a single workspace at a time at the
  Garden level; no backlog entry.
- Provider-instance pooling beyond simple LRU-of-one — the cache
  intentionally holds at most one provider; an eviction event is a
  full reconstruction, which is cheap relative to the LLM call. No
  backlog entry.
