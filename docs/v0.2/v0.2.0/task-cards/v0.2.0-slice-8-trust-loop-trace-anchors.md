## 1. Background & Goal

**Background.** The Trustworthy Memory Loop spans seven steps:
recall delivery â†’ usage receipt â†’ candidate signal â†’ proposal â†’
accepted proposal â†’ durable application â†’ post-apply recall. Today
the EventLog records each step but a downstream auditor cannot join
the chain from the daemon alone: `soul.signal.emitted`
(`packages/protocol/src/events/signal.ts:25`) and the proposal
events (`memory-governance.ts`, around lines 120-160) carry no
backreference to the `delivery_id` that started the chain. An
auditor must reconstruct causality through entity ids and run ids,
which is fragile.

Note the real code paths: the `soul.propose_memory_update` /
`soul.review_memory_proposal` MCP handlers in
`apps/core-daemon/src/mcp-memory-tool-handler.ts` (around lines
500-520) only route â€” the actual proposal-event construction and
persistence live in `apps/core-daemon/src/mcp-memory-proposal-workflow.ts`
(`proposal.created` built around lines 225-236, `proposal.resolved`
around lines 342-359), and the row is written through
`packages/storage/src/repos/proposal-repo.ts`
(`ProposalCreateInput` / `createProposalWithEvents`).
`packages/core/src/proposal-service.ts` is **not** on the live MCP
persistence path. Also note: the signal source enum
(`packages/protocol/src/candidate-memory-signal.ts:41-46`) has no
`agent_proposed` value â€” the agent-originated source is
`SignalSource.MODEL_TOOL`; `GARDEN_COMPILE` / `USER_SEED` / `IMPORT`
are the other sources.

**Goal.** Add an optional `source_delivery_ids` array (min 1) to the
signal and proposal payloads, add the same optional
`source_delivery_ids` array field to the `soul.emit_candidate_signal`
and `soul.propose_memory_update` request schemas (a single delivery
is just a one-element array â€” there is no separate singular
`delivery_id` request field, which avoids any "both present"
precedence ambiguity), thread it through the emit handler and the
proposal workflow, persist it on the proposal row via a new nullable
column (migration `066`), echo it on `proposal.resolved`, and emit a
daemon-side WARN when a `MODEL_TOOL`-sourced signal is missing the
anchor. A producer-set test pins the known signal-emitting paths so
a new
emitter cannot be added without an anchor-handling decision.

## 2. Allowed Scope

### packages/protocol/src/events/memory-governance.ts (around lines 120-160)

- **Change**: locate `SoulProposalCreatedPayloadSchema` and
  `SoulProposalResolvedPayloadSchema`. If `proposal.created`
  currently re-exports the shared base object schema, change it to
  `.extend({ source_delivery_ids: ... })` so the field rides on
  `proposal.created` independently of `proposal.resolved`. Add the

  (resolved echoes the persisted value).
- Verify the shared `TransitionEventPayloadObjectSchema` /
  `MemoryGovernanceObjectPayloadObjectSchema` composition still
  type-checks after the `.extend` change.

### packages/protocol/src/soul/mcp-types.ts (MCP request schemas)

- **Hazard â€” do NOT extend the shared candidate-signal schema.**
  `McpEmitCandidateSignalRequestSchema` (the `soul.emit_candidate_signal`
  request) is also embedded as the element schema inside
  `GardenTaskResultEnvelopeSchema.candidate_signals` (the
  `garden.complete_task` payload). Adding the anchor to that shared
  schema would let a Garden host worker submit a recall anchor
  through `garden.complete_task` â€” but Garden-originated extraction
  has no prior recall delivery, so that is a trust-boundary leak
  (forged or ambiguous audit attribution).
- **Change**: split the schemas:
  - the candidate-signal **content** element used inside
    `GardenTaskResultEnvelopeSchema.candidate_signals` stays
    **anchor-free** (no `source_delivery_ids`)
  - introduce a distinct request schema for
    `soul.emit_candidate_signal` that carries an optional
    `source_delivery_ids: NonEmptyStringSchema.array().min(1).optional()`
    as a **top-level request field** alongside the candidate-signal
    content â€” not inside the content object. There is **no** separate
    singular `delivery_id` request field; a single delivery is a
    one-element array. An empty array is rejected (`.min(1)`).
  - for `soul.propose_memory_update`: add the same optional
    `source_delivery_ids` array field to its request schema. Verify
    first whether that request schema is shared with any non-MCP
    path; if it is, apply the same split.
  - `soul.review_memory_proposal` needs no new input field â€” the
    value is read back from the proposal row on resolve.
  All additions are additive minors under Â§25. The MCP catalog's
  `inputSchema` derives from these zod schemas via
  `zod-to-json-schema`, so the new optional field appears in the
  published catalog automatically â€” no manual catalog edit, but the
  catalog snapshot test (if one exists) is updated.

### apps/core-daemon/src/mcp-memory-tool-handler.ts (emit handler + routing)

- **`soul.emit_candidate_signal` handler**: read the optional
  `source_delivery_ids` array from the validated input; when present
  and non-empty, pass it to the signal-emit path so it lands as
  `source_delivery_ids` on the `soul.signal.emitted` event. The emit
  path currently constructs the signal with `SignalSource.MODEL_TOOL`
  â€” keep that.
- **`soul.propose_memory_update` routing**: pass the optional
  `source_delivery_ids` array through to
  `mcp-memory-proposal-workflow.ts`.
- **`soul.review_memory_proposal` routing**: no input change; the
  workflow reads `source_delivery_ids` from the proposal row.
- **WARN guard**: when emitting `soul.signal.emitted` with
  `source === SignalSource.MODEL_TOOL` and no `source_delivery_ids`,
  log a daemon-side WARN with the signal id and source. Signals with
  source `GARDEN_COMPILE` / `USER_SEED` / `IMPORT` do not trigger
  the warning.

### apps/core-daemon/src/mcp-memory-proposal-workflow.ts

- **Change**: `propose_memory_update` accepts the optional
  `source_delivery_ids` array, threads it into the `proposal.created`
  event payload (around lines 225-236) and into the
  `ProposalCreateInput` so it persists on the row.
- **Change**: `review_memory_proposal` reads `source_delivery_ids`
  from the loaded proposal row and writes it into the
  `proposal.resolved` payload (around lines 342-359).

### packages/storage/src/repos/proposal-repo.ts

- **Change**: `ProposalCreateInput` gains an **optional** property
  `source_delivery_ids?: readonly string[] | null`. The row INSERT
  via `createProposalWithEvents` normalizes a missing value
  (`undefined`) to `null` before writing; the rowâ†’object mapping
  reads it back. Serialize as JSON text in the SQLite column (the
  repo already uses JSON columns for similar list fields â€” follow
  the existing pattern, do not introduce a new serialization
  convention).
- **Why optional, not required**: the proposal-create path has
  other production callers besides the MCP workflow â€”
  `packages/core/src/proposal-service.ts` (around line 217),
  `apps/core-daemon/src/budget-wiring.ts` (around line 49), and
  repo tests. Making the field required would force edits to all of
  them for no behavioural reason. Optional + repo-side
  `undefined â†’ null` normalization means those callers compile
  unchanged and persist a NULL anchor (which is correct â€” they are
  not on a recall-attributable chain).

### packages/storage/src/migrations/066-proposal-source-delivery-ids.sql (new)

- **Change**: `ALTER TABLE proposals ADD COLUMN source_delivery_ids
  TEXT;` (nullable; existing rows get NULL). Confirm `066` is the
  next free number after the current `065-proposal-target-baseline.sql`
