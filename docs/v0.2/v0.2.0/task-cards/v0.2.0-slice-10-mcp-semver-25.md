## 1. Background & Goal

**Background.** v0.2.0 adds public-surface fields — all additive,
all optional or defaulted:
- `pressure_ratio` (optional / `.default(0)`) on `BudgetSnapshot`
  (`packages/protocol/src/soul/budget-snapshot.ts`)
- `host_context` (optional) on the `soul.recall` MCP input
- `domain_weight_overrides` (optional) on `RecallPolicy`
- `resolved_activation_weights` (optional) on `RecallScoreFactors`
  (`packages/protocol/src/soul/recall-candidate.ts`) — exposed via
  `soul.recall` → `MemorySearchResultSchema.score_factors`
- `source_delivery_ids` (optional array, min 1) on three EventLog
  payloads (`soul.signal.emitted`, `soul.proposal.created`,
  `soul.proposal.resolved`)
- `source_delivery_ids` (optional array, min 1) on two MCP request
  inputs (`soul.emit_candidate_signal`, `soul.propose_memory_update`;
  `soul.review_memory_proposal` reads the stored value, no new input)

It removes internal-only types (`ConversationProvider` and friends —
no MCP / EventLog / config surface, out of §25 scope). A sibling
memory-first agent
will eventually pin `@do-soul/alaya-protocol` and follow Alaya's
releases. There is currently no written SemVer policy, so a
downstream consumer cannot tell which changes are safe to follow
and which require a migration.

**Goal.** Write invariant §25 (MCP and Protocol SemVer Contract)
into `docs/handbook/invariants.md`; open the deprecation-tracking
section in `docs/handbook/maintenance.md`; mark `#BL-008` Resolved
in `docs/handbook/backlog.md`; update
`docs/handbook/runtime-status.md` readiness rows; write the v0.2.0
release notes.

## 2. Allowed Scope

### docs/handbook/invariants.md (new §25)

- **Change**: append a new numbered section §25 with this content
  (adjust numbering if §25 is taken):

  ```
  25. MCP and Protocol SemVer Contract. Alaya publishes three
  concentric public contracts that downstream consumers (sibling
  agents, MCP clients, SDK pinning) MAY rely on:

  (a) the MCP tool surface — tool names and descriptions in
      packages/engine-gateway/src/provider/soul-tool-specs.ts, AND
      every Zod schema TRANSITIVELY REACHABLE from an MCP request or
      response type defined in packages/protocol/src/soul/mcp-types.ts
      (the request schemas and the response/result schemas, plus
      every schema they compose, no matter which file or directory
      those schemas live in). soul-tool-specs.ts holds only tool
      names + parametersSchema; the request and result shapes live
      in mcp-types.ts and the schemas it composes — so all of those
      are part of the contract. The AUTHORITATIVE definition is
      transitive reachability, not a hand-maintained file list; the
      semver-surface.test.ts snapshot (below) computes the reachable
      set and pins it, so a newly-reached schema is caught
      automatically. Files in the reachable set today include (this
      enumeration is illustrative, NOT exhaustive — the snapshot is):
      mcp-types.ts, recall-candidate.ts (RecallScoreFactorsSchema
      via MemorySearchResultSchema), memory-entry.ts, recall-policy.ts,
      budget-snapshot.ts, packages/protocol/src/candidate-memory-signal.ts
      (this one lives directly under packages/protocol/src/, not
      under soul/), packages/protocol/src/schema-primitives.ts,
      packages/protocol/src/soul/memory-graph.ts,
      packages/protocol/src/soul/object-kind.ts, and
      packages/protocol/src/soul/proposal.ts.
  (b) the EventLog payload schemas under packages/protocol/src/
      events/*;
  (c) the runtime control-plane config schemas under packages/
      protocol/src/app-config.ts.

  Workspace-internal interfaces (cross-package TypeScript types with
  no MCP / EventLog / config surface, e.g. engine-gateway provider
  abstractions before any consumer exists) are out of SemVer scope
  until they have at least one production consumer.

  - Patch (x.y.Z): bug fixes, performance, internal refactor. No
    schema change.
  - Minor (x.Y.z): additive only. New optional payload/result
    fields, new event types, new tool names, new enum values not in
    discriminator keys. Required fields, narrowed nullables, removed
    enum values, renamed tools, semantic redefinitions of an
    existing field are NOT minor.
  - Major (X.y.z): any removal, rename, or semantic redefinition on
    any of the three layers.

  Deprecation policy. Removing a public symbol requires (i)
  @deprecated JSDoc on the schema at least one minor release before
  removal, (ii) a maintenance entry in docs/handbook/maintenance.md
  naming the symbol and target removal version, (iii) a
  sibling-compat smoke test asserting the old shape still parses.
  Earliest removal is N+1 minor.

  Downstream pinning. Sibling consumers pin @do-soul/alaya-protocol
  to the minor (^0.2.0). The MCP tool surface pins via the protocol
  package version, since the tool request/response schemas live in
  @do-soul/alaya-protocol. Cross-major upgrades require reading the
  docs/handbook/maintenance.md migration entry.

  PRs touching packages/protocol/src/events/, packages/protocol/
  src/app-config.ts, packages/engine-gateway/src/provider/soul-tool-specs.ts,
  or any Zod schema transitively reachable from an MCP request or
  response type in mcp-types.ts (the semver-surface.test.ts snapshot
  is the source of truth for the current set — if your change makes
  that snapshot move, you are touching the MCP contract) MUST cite
  §25 and declare the SemVer step.
  ```

### Optional: SemVer lint test

- **Change (recommended; treat as required for v0.2.0 since it is
  what makes §25(a) self-policing)**: a test under
  `packages/protocol/src/__tests__/semver-surface.test.ts` that
  snapshots: (i) the set of MCP tool names from `soul-tool-specs.ts`;
  (ii) **the transitive reachability closure of every MCP request
  and response Zod schema declared in `mcp-types.ts`** — walk each
  schema's `.shape` (and `.unwrap()` / array element / union member /
  record value) recursively, collecting `(schemaName, fieldKey)`
  pairs and the source module of each named schema; this is the
  authoritative "MCP contract surface" and it automatically picks up
  `recall-candidate.ts`'s `RecallScoreFactorsSchema`,
  `schema-primitives.ts` primitives like `BoundedIdSchema`,
  `soul/memory-graph.ts`, `soul/object-kind.ts`, `soul/proposal.ts`,
  `candidate-memory-signal.ts`, etc., without anyone maintaining a
  file list; (iii) the set of exported EventLog payload schema keys
  from `packages/protocol/src/events/*`; (iv) the runtime config
  schema keys from `app-config.ts`. The test fails if any name/key
  disappears from any snapshot (or if a *new module* enters the MCP
  reachability closure) without an accompanying snapshot update —
  turning §25's "removal is major" and "the contract surface grew"
  rules into tripwires. The snapshot is committed; the test
  instructions explain that an intentional removal updates the
  snapshot in the same PR that
  bumps the major.

## 3. Deferred

- Automated SemVer enforcement via a heavyweight tool like
  `api-extractor` diffing public `.d.ts` is deferred; the
  `semver-surface.test.ts` reachability snapshot above is the
  v0.2.0 tripwire and is in-scope for this card (not optional). A
  fuller CI gate can be added when the surface stabilizes. No
  backlog entry.
- The sibling-compat smoke test mentioned in §25's deprecation
  policy is a no-op today (nothing is deprecated). The first actual
  deprecation in a future release brings the first smoke test. No
  backlog entry.
