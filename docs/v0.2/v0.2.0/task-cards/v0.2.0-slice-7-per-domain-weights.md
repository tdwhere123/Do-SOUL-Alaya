## 1. Background & Goal

**Background.** `activation_weights_phase4b`
(`packages/protocol/src/soul/dynamics-constants.ts:22-31`) is a
single global constant: `scope_match=0.18`, `domain_match=0.18`,
`retention=0.18`, `freshness=0.16`, `relevance=0.1`,
`graph_support=0.05`, `budget_penalty=0.1`,
`conflict_penalty=0.05`. The only domain handling today is the
filter at `deterministic_match.domain_tag_filter` — a domain either
participates or doesn't; the weights are uniform across all
domains.

Workspaces have genuinely different recall needs: a
project-document workspace may want `retention` to dominate; a
research-notes workspace may want `relevance` higher. v0.2.0
exposes optional per-domain weight overrides on the policy.

**Goal.** Add `domain_weight_overrides` to `RecallPolicy`. Resolve
overrides per memory entry by intersecting its `domain_tags` with
the override map. Validate every merged weight set still
sums-to-one via `assertActivationWeightsSumToOne`. Record the
resolved weight set in `RecallScoreFactorsSchema` — which is a
**public MCP output** (`soul.recall` returns it through
`MemorySearchResultSchema.score_factors`), so this addition is an
additive minor under §25 and must be reflected in Slice 10's §25
inventory and the MCP catalog snapshot — so an auditor (or a
sibling agent) can reproduce a score after the fact.

## 2. Allowed Scope

### packages/protocol/src/soul/recall-policy.ts

- **Change**: add
  ```ts
  domain_weight_overrides: z
    .record(NonEmptyStringSchema, ActivationWeightsPatchSchema)
    .optional()
  ```
  to `RecallPolicySchema` at line 53.
- **Define** `ActivationWeightsPatchSchema` as a `Partial<...>` of
  `activation_weights_phase4b` keys, where every value is a
  `z.number().min(0).max(1)`. The schema does NOT validate
  sum-to-one at parse time — the merge-time guard
  (`assertActivationWeightsSumToOne`) does that after composition
  with the base weights.
- The `.optional()` ensures storage-side `RecallPolicy` rows from
  v0.1 still parse with no migration.

### packages/core/src/recall-service.ts (lines 936-987)

- **Change**: inside `computeEffectiveScoreDetails`, before applying
  weights, resolve the effective weight set:
  - if `policy.domain_weight_overrides` is undefined, return
    `DYNAMICS_CONSTANTS.activation_weights_phase4b` unchanged
  - else, intersect `entry.domain_tags` with the override map keys
  - if the intersection is empty, return base weights unchanged
  - if multiple keys match, deterministic priority: sort the
    matched keys by string `localeCompare`, take the first; the
    matched override's patch is merged onto the base weights
  - call `assertActivationWeightsSumToOne(merged)`; on failure,
    log a daemon-side ERROR with the policy ID and the domain tag,
    and fall back to base weights (fail-closed but not
    fail-stop — recall stays available)
  - populate the resolved weight set into the entry's
    `RecallScoreFactors` (see next section).

### packages/protocol/src/soul/recall-candidate.ts (`RecallScoreFactorsSchema`)

- **Change**: extend `RecallScoreFactorsSchema` with a new
  **optional** field — e.g.
  `resolved_activation_weights: ActivationWeightsSchema.optional()`
  (the full eight-key weight set actually used to score this entry,
  after any per-domain override merge). It is optional so existing
  recall responses / consumers that don't expect it still validate;
  the daemon always populates it going forward.
- **This is a public surface change.** `RecallScoreFactorsSchema` is
  returned by `soul.recall` via `MemorySearchResultSchema.score_factors`
  in `packages/protocol/src/soul/mcp-types.ts`, so the addition is an
  additive minor under §25. It MUST appear in Slice 10's §25 / release
  inventory and the MCP catalog snapshot must be regenerated. Do not
  treat it as an internal-only type change.

### packages/core/src/recall-service-helpers.ts (`assertActivationWeightsSumToOne`)

- **Locate** (line 349 per plan).
- **Change**: ensure the function tolerates `Partial<...>` inputs
  by treating missing keys as their base value before checking the
  sum. The check is `Math.abs(sum - 1) < 1e-6`. Document the
  tolerance in a code comment.

### Tests

- **Add**:
  - `RecallPolicy` parses without `domain_weight_overrides` (v0.1
    rows)
  - `RecallPolicy` parses with one override
  - merge resolution: an entry tagged `["work"]` matches an override
    keyed `"work"`; resolved weights include the override's diff
  - multi-tag tie-break: an entry tagged `["work", "research"]`
    with overrides for both prefers `"research"` (alphabetical
    first)
  - sum-to-one violation: an override that breaks the invariant
    logs and falls back to base weights without throwing to the
    caller
  - no-override path: an entry with no tag overlap returns base
    weights, byte-identical scoring to v0.1 — pinned by a fixture

## 3. Deferred

- Per-domain `budget_penalty` shape (not just the weight, but the
  curve from Slice 5) — deferred indefinitely; v0.2.0 keeps the
  penalty curve global. No backlog entry until concrete need
  surfaces.
- Override audit history (who changed what override, when) —
  deferred to a follow-up release; today overrides are durable in
  the policy but mutation history is in EventLog through normal
  policy-update events. No backlog entry.
