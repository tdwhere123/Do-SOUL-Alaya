## 1. Background & Goal

**Background.** Slice 8 added `source_delivery_ids` to the signal
and proposal payloads, an optional top-level `source_delivery_ids`
array (min 1) input to `soul.emit_candidate_signal` /
`soul.propose_memory_update` (a single delivery is a one-element
array; there is no singular `delivery_id` request field), and
threaded it through the emit handler and the proposal workflow
(`apps/core-daemon/src/mcp-memory-proposal-workflow.ts` +
`packages/storage/src/repos/proposal-repo.ts`, migration `066`). The
plan's success criterion for "Trustworthy Memory Loop daemon-only
auditable" is a test that proves an auditor can reconstruct the
causal chain from the EventLog table alone, joining solely on
`delivery_id` — no entity ids, no run ids.

**Goal.** Add an integration test that drives the full agent-driven
loop through the MCP handlers against an in-memory SQLite EventLog,
then asserts a single SQL query — joining solely on
`source_delivery_ids` array membership ↔ the `delivery_id` scalar,
with no `entity_id` / `run_id` predicate — reconstructs the
five-event chain in causal order (Case 1), and that the same
membership join works for a multi-delivery proposal created through
the public array request (Case 1b). Add a second case for the
Garden-originated path that proves only the two guarantees the
v0.2.0 public surface can express: a `GARDEN_COMPILE` signal carries
no `source_delivery_ids`, and the WARN guard does not fire for it.
Case 2 does NOT claim a causal join or a "three-event chain" for the
Garden path — v0.2.0 leaves Garden extraction intentionally
unanchored, and the public `soul.propose_memory_update` request has
no signal-id field to link on.

## 2. Allowed Scope

### apps/core-daemon/src/__tests__/trustworthy-loop-trace.test.ts (new)

- **Change**: a new test file using the existing daemon test harness
  (in-memory `better-sqlite3`, the MCP SDK in-memory transport, the
  same bootstrap pattern as
  `gate4-attached-agent-mcp-proof.test.ts`).

- **Case 1 — agent-driven chain (single delivery):**
  1. `soul.recall` with a synthetic query → capture `delivery_id`
     from the `soul.recall.delivered` EventLog row.
  2. `soul.report_context_usage` with `usage_state = "used"` for
     that delivery → `soul.context_usage.reported` row.
  3. `soul.emit_candidate_signal` with `source_delivery_ids: [deliveryId]`
     in the input → `soul.signal.emitted` row with
     `source_delivery_ids = [deliveryId]`.
  4. `soul.propose_memory_update` with `source_delivery_ids: [deliveryId]`
     → proposal row persists it; `soul.proposal.created` row carries it.
  5. `soul.review_memory_proposal` (accept or reject — either
     resolves) → `soul.proposal.resolved` row echoes
     `source_delivery_ids`.
  6. **Assertion**: a single SQL query against the `event_log`
     table reconstructs the chain using ONLY delivery-id linkage —
     no `entity_id` or `run_id` predicate. Because
     `source_delivery_ids` is a JSON **array**, the join must be by
     array membership, not by reading the array as a scalar:
     expand it with `json_each(payload_json, '$.source_delivery_ids')`
     and join each member to the `delivery_id` scalar
     (`payload_json ->> '$.delivery_id'`) on the delivered / reported
     rows. The query returns exactly five rows in causal order by
     the EventLog's monotonic ordering column: `soul.recall.delivered`
     → `soul.context_usage.reported` → `soul.signal.emitted` →
     `soul.proposal.created` → `soul.proposal.resolved`. The test
     also inspects the SQL string and asserts it contains no
     `entity_id` / `run_id` predicate and no `->> '$.source_delivery_ids'`
     scalar comparison (it must use `json_each` membership).

- **Case 1b — agent-driven chain (multi-delivery, via the public
  request path):** same as Case 1, but two recall deliveries are
  made (`d1`, `d2`), and the proposal is created with
  `source_delivery_ids: [d1, d2]` **through the public
  `soul.propose_memory_update` request** (not by seeding a row
  directly — the point is to exercise the public array API). The
  same `json_each(payload_json, '$.source_delivery_ids')` membership
  query, filtered to either `d1` or `d2`, returns the proposal's
  `created` / `resolved` rows — proving the join is membership-based
  and cannot rely on serialized-array equality, and that the public
  request actually accepts and persists the multi-element array.

- **Case 2 — Garden-originated signal (negative / non-anchored
  case):** the v0.2.0 public request shapes carry no signal id
  (`soul.propose_memory_update` has only `target_object_id` /
  `proposed_changes` / `reason`, and the workflow sets
  `proposal.derived_from` from `target_object_id`), so a
  Garden-originated signal cannot be causally linked to a downstream
  proposal in the EventLog — and that is fine: v0.2.0 deliberately
  leaves Garden-originated extraction unanchored. Case 2 therefore
  proves only the two guarantees that ARE expressible:
  1. Drive a `POST_TURN_EXTRACT` Garden task to completion (the
     existing test harness already does a Garden background pass).
     The extracted candidate produces a `soul.signal.emitted` row
     with source `GARDEN_COMPILE`.
  2. **Assertion**: that signal row's payload has **no**
     `source_delivery_ids` key (or it is NULL) — Garden signals are
     never anchored.
  3. **Assertion**: the daemon WARN guard from Slice 8 did NOT fire
     for that `GARDEN_COMPILE` signal (capture the logger; assert no
     WARN with that signal id).
  4. (Optional, illustrative not contractual) the test may still run
     a `soul.propose_memory_update` / `soul.review_memory_proposal`
     on a fresh `target_object_id` and assert those proposal events
     also carry no `source_delivery_ids` — but it does NOT claim a
     causal join from the Garden signal to that proposal, because no
     field links them. Do not assert "three rows in causal order"
     for the Garden path.

- **No production code change** in this card. If Case 1 / 1b reveals
  a threading gap, that gap is fixed in Slice 8 (re-open it) — this
  card stays test-only. If a reviewer decides Garden signal →
  proposal reconstruction IS required, that is a new design decision
  (a non-delivery causal field on the proposal request + payload),
  out of scope for this card; open a backlog issue rather than
  widening Slice 9.
