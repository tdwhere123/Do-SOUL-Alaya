# Alaya Memory — Benchmark Baseline (2026-06-15)

Durable baseline for future recall/QA research. All numbers are **real full-dataset
runs** (not subset-weighted): LongMemEval-S = all 500 questions, LoCoMo = all 10
conversations. `Overall` is the micro-average over all questions, computed directly.

## Configuration

| Knob | Value |
|---|---|
| Recall | **3b-on** (`ALAYA_EVIDENCE_FULL_TURN`, now the product default — see PR) |
| Answer model | `gpt-5.4-mini` (fixed; operator constraint) |
| Judge model | `gpt-4o-2024-11-20` + official LongMemEval per-type rubric prompt |
| Embedding | local ONNX `paraphrase-multilingual-MiniLM-L12-v2` |
| Delivery | natural recall top-K (no agent-side LLM filter — that is the consuming agent's job) |

## Judge methodology (read before comparing to other systems)

The LLM-judge is a **measurement instrument**, not part of the product. Two findings
fix how it must be read:

1. **The judge only matters for `single-session-preference`.** That category is scored
   against a subjective *rubric*; the other 5 categories are scored "does the response
   contain the correct answer" and are **judge-invariant** (gpt-4o == gpt-5.4-mini to
   within ±3pp, confirmed this session by re-judging the same answers with both models).

2. **On preference, the judge model must follow the *official* (deliberately lenient)
   rubric.** The official LongMemEval preference rubric says "the model does not need to
   reflect all the points… correct as long as it recalls and utilizes the user's personal
   information." gpt-4o follows this faithfully (validated by gold adjudication of every
   mini-vs-gpt-4o disagreement, and by a scrambled-gold control: 0% false-yes). A stronger
   model (gpt-5.4-mini, Intelligence Index 49 vs gpt-4o's 17) imposes *extra* strictness it
   was not asked for, deflating preference ~20-30pp — this is a **prompt-calibration issue,
   not a capability issue**: with an explicit lenient-rubric clause added to the judge
   prompt (V2), gpt-5.4-mini's agreement with gpt-4o rises 73%→90% while the scrambled-gold
   control stays 0%. The V2 calibrated prompt is landed for future unified baselines; this
   baseline reports the gpt-4o official-rubric numbers for comparability with published
   LongMemEval results.

Answer = gpt-5.4-mini, Judge = gpt-4o (official rubric), Recall = 3b-on, full 500.

|---|---|---|---|---|---|---|---|
| Alaya | **87.1** | 85.7 | 80.0 | 51.9 | 73.1 | 64.7 | **69.0** |

knowledge-update 57/78, temporal 86/133; overall 345/500.

> **No category reaches ≥90 on the full set.** Closest: ss-user 87.1, ss-assistant 85.7.
> (Earlier n=35-subset numbers — e.g. ss-user 94.3 — were optimistic samples; these are the
> real full-dataset values.)
>
> preference **+ agent-side LLM selection** (the consuming agent's job, not Alaya core;
> bench simulates the agent picking from the wide pool): **96.7** (29/30, full category).
> Shown separately because Alaya delivers the candidate pool and the agent does final
> selection — see the §D / filter discussion in the worklog.

## ② LoCoMo — LLM-judge accuracy (%)

Answer = gpt-5.4-mini, Judge = gpt-4o, full 10 conversations.

| System | single-hop | multi-hop | temporal | open-domain | adversarial | **Overall** |
|---|---|---|---|---|---|---|
| Alaya | 14.5 | 13.0 | 50.2 | 53.0 | 18.2 | **37.4** |

n = 1982 questions over all 10 conversations. single-hop 41/282, multi-hop 12/92,
temporal 161/321, open-domain 446/841, adversarial 81/446; overall 741/1982.

> LoCoMo is **recall-bound** (R@5 = 46.7%, see below) — far below LongMemEval. The
> low single-hop/multi-hop are dominated by recall misses + multi-item list questions
> scored by strict subset (a model that answers 1 of 2 listed items is judged wrong).
> cat-5 (adversarial) requires abstention scoring the answer model is not prompted for;
> reported as-is with that caveat. Raising LoCoMo needs a dedicated recall investigation
> (3b and a retrieval-tuned embedding were both ~flat on LoCoMo recall — see worklog).

## Supplementary — retrieval metrics (not the headline; recall-only)

`any-gold@5` = ≥1 gold memory in top-5. `full-gold@5` = ALL gold memories in top-5
(meaningful only for the multi-gold categories: knowledge-update, multi-session, temporal).

### LongMemEval-S

Overall (n=500): **any-gold@5 = 86.8%**, full-gold@5 = 20.3% (from pool dump; gold-in-pool 89.7%).

|---|---|---|---|---|

| single-session-assistant | (multi-mem) | 90.6 | 90.6 | 71.7 |
| single-session-preference | ~1-2 | 66.7 | 63.3 | 13.3 |

## Diagnostic — oracle ceiling (NOT a score; isolates reader from recall)

`oracle` = deliver the correct gold verbatim (skip recall) and ask the answer model. It is
the **answer-model reading-comprehension upper bound** with perfect recall. Gap from the QA
number to oracle = what recall can still recover; gap from oracle to 100% = answer-model
limit that recall cannot fix.

| Category | oracle (gpt-4o judge) |
|---|---|
| single-session-user | 94.3 |
| single-session-preference | 96.7 |
| single-session-assistant | 88.6 |
| knowledge-update | 85.7 |
| temporal-reasoning | 80.0 |
| multi-session | 62.9 |

(oracle from this session's per-category runs; gpt-4o judge.)

## Why not ≥90 everywhere? (summary; full divergent analysis in the worklog)

Headline: **LongMemEval-S 69.0% overall, LoCoMo 37.4% overall; no category ≥90 on full data.**
Two distinct walls:

- **Reader wall (LongMemEval factual cats)**: ss-assistant 85.7, knowledge-update 73.1,
  temporal 64.7, multi-session 51.9. any-gold@5 is high (90-97%) so the gold is usually
  retrieved, but (a) **full-gold@5 is ~6-9%** for the multi-gold cats — not all evidence
  fits in the delivery window — and (b) the answer model's oracle ceiling (perfect recall,
  n=35 subset) is itself <90 (multi-session ~63, temporal ~80) — reading comprehension
  (enumeration, date-grounding, paraphrase), which recall cannot fix. multi-session is the
  hardest on both axes.
- **§D recall wall (preference)**: any-gold@5 only 63% — query words are disjoint from the
  gold preference fact. 3b + judge calibration land it at 80.0 natural; the agent-side LLM
  selection takes it to 96.7. The fusion-layer fix (direction a) was investigated and found
  **not landable standalone** (see worklog) — it needs a retrieval-tuned embedding to have
  any signal.
- **LoCoMo**: recall-bound (R@5 = 46.7%); dedicated recall investigation needed (3b and a
  retrieval-tuned embedding were both ~flat on LoCoMo recall).
